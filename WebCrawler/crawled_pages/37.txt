<p>The <b>Secure Shell Protocol</b> (<b>SSH Protocol</b>) is a <a href="/wiki/Cryptographic" class="mw-redirect" title="Cryptographic">cryptographic</a> <a href="/wiki/Network_protocol" class="mw-redirect" title="Network protocol">network protocol</a> for operating <a href="/wiki/Network_service" title="Network service">network services</a> securely over an <a href="/wiki/Unsecured_network" class="mw-redirect" title="Unsecured network">unsecured network</a>.<sup id="cite&#95;ref-rfc4251&#95;1-0" class="reference"><a href="#cite_note-rfc4251-1"><span class="cite-bracket">&#91;</span>1<span class="cite-bracket">&#93;</span></a></sup> Its most notable applications are <a href="/wiki/Remote_login" class="mw-redirect" title="Remote login">remote login</a> and <a href="/wiki/Command-line" class="mw-redirect" title="Command-line">command-line</a> execution.
</p><p>SSH was designed for <a href="/wiki/Unix-like" title="Unix-like">Unix-like</a> operating systems as a replacement for <a href="/wiki/Telnet" title="Telnet">Telnet</a> and <a href="/wiki/Computer_security" title="Computer security">unsecured</a> remote <a href="/wiki/Unix_shell" title="Unix shell">Unix shell</a> protocols, such as the Berkeley <a href="/wiki/Remote_Shell" title="Remote Shell">Remote Shell</a> (rsh) and the related <a href="/wiki/Rlogin" class="mw-redirect" title="Rlogin">rlogin</a> and <a href="/wiki/Rexec" class="mw-redirect" title="Rexec">rexec</a> protocols, which all use insecure, <a href="/wiki/Plaintext" title="Plaintext">plaintext</a> methods of authentication, such as <a href="/wiki/Password" title="Password">passwords</a>.
</p><p>Since mechanisms like <a href="/wiki/Telnet" title="Telnet">Telnet</a> and <a href="/wiki/Remote_Shell" title="Remote Shell">Remote Shell</a> are designed to access and operate remote computers, sending the <a href="/wiki/Authentication" title="Authentication">authentication</a> tokens (e.g. <a href="/wiki/Username" class="mw-redirect" title="Username">username</a> and <a href="/wiki/Password" title="Password">password</a>) for this access to these computers across a <a href="/wiki/Computer_network" title="Computer network">public network</a> in an unsecured way poses a great risk of third parties obtaining the password and achieving the same level of access to the remote system as the telnet user. Secure Shell mitigates this risk through the use of <a href="/wiki/Encryption" title="Encryption">encryption</a> mechanisms that are intended to hide the contents of the transmission from an observer, even if the observer has access to the entire <a href="/wiki/Data_stream" title="Data stream">data stream</a>.<sup id="cite&#95;ref-2" class="reference"><a href="#cite_note-2"><span class="cite-bracket">&#91;</span>2<span class="cite-bracket">&#93;</span></a></sup>
</p><p>Finnish computer scientist Tatu Ylönen designed SSH in 1995 and provided an implementation in the form of two commands, <style data-mw-deduplicate="TemplateStyles:r886049734">.mw-parser-output .monospaced{font-family:monospace,monospace}</style><span class="monospaced">ssh</span> and <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r886049734" /><span class="monospaced">slogin</span>, as secure replacements for <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r886049734" /><span class="monospaced">rsh</span> and <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r886049734" /><span class="monospaced">rlogin</span>, respectively. Subsequent development of the protocol suite proceeded in several developer groups, producing several variants of implementation. The protocol specification distinguishes two major versions, referred to as SSH-1 and SSH-2. The most commonly implemented software stack is <a href="/wiki/OpenSSH" title="OpenSSH">OpenSSH</a>, released in 1999 as open-source software by the <a href="/wiki/OpenBSD" title="OpenBSD">OpenBSD</a> developers. Implementations are distributed for all types of operating systems in common use, including <a href="/wiki/Embedded_system" title="Embedded system">embedded systems</a>.
</p><p>SSH applications are based on a <a href="/wiki/Client%E2%80%93server_architecture" class="mw-redirect" title="Client–server architecture">client–server architecture</a>, connecting an <a href="/wiki/SSH_client" class="mw-redirect" title="SSH client">SSH client</a> instance with an <a href="/wiki/SSH_server" class="mw-redirect" title="SSH server">SSH server</a>.<sup id="cite&#95;ref-rfc4252&#95;3-0" class="reference"><a href="#cite_note-rfc4252-3"><span class="cite-bracket">&#91;</span>3<span class="cite-bracket">&#93;</span></a></sup> SSH operates as a layered protocol suite comprising three principal hierarchical components: the <i>transport layer</i> provides server authentication, confidentiality, and integrity; the <i>user authentication protocol</i> validates the user to the server; and the <i>connection protocol</i> <a href="/wiki/Multiplexing" title="Multiplexing">multiplexes</a> the encrypted tunnel into multiple logical communication channels.<sup id="cite&#95;ref-rfc4251&#95;1-1" class="reference"><a href="#cite_note-rfc4251-1"><span class="cite-bracket">&#91;</span>1<span class="cite-bracket">&#93;</span></a></sup>
</p>
<meta property="mw:PageProp/toc" />
<div class="mw-heading mw-heading2"><h2 id="Definition">Definition</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Secure_Shell&amp;action=edit&amp;section=1" title="Edit section: Definition"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>SSH uses <a href="/wiki/Public-key_cryptography" title="Public-key cryptography">public-key cryptography</a> to <a href="/wiki/Authenticate" class="mw-redirect" title="Authenticate">authenticate</a> the remote computer and allow it to authenticate the user, if necessary.<sup id="cite&#95;ref-rfc4252&#95;3-1" class="reference"><a href="#cite_note-rfc4252-3"><span class="cite-bracket">&#91;</span>3<span class="cite-bracket">&#93;</span></a></sup>
</p><p>SSH may be used in several methodologies. In the simplest manner, both ends of a communication channel use automatically generated public-private key pairs to encrypt a network connection, and then use a <a href="/wiki/Password" title="Password">password</a> to authenticate the user.
</p><p>When the public-private key pair is generated by the user manually, the authentication is essentially performed when the key pair is created, and a session may then be opened automatically without a password prompt. In this scenario, the public key is placed on all computers that must allow access to the owner of the matching private key, which the owner keeps private. While authentication is based on the private key, the key is never transferred through the network during authentication. SSH only verifies that the same person offering the public key also owns the matching private key.
</p><p>In all versions of SSH, it is important to verify unknown <a href="/wiki/Public_key" class="mw-redirect" title="Public key">public keys</a>, i.e., <a href="/wiki/Public-key_cryptography#Associating_public_keys_with_identities" title="Public-key cryptography">associate the public keys with identities</a>, before accepting them as valid. Accepting an attacker's public key without validation will authorize an unauthorized attacker as a valid user.
</p>
<div class="mw-heading mw-heading2"><h2 id="Authentication:_OpenSSH_key_management">Authentication: OpenSSH key management</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Secure_Shell&amp;action=edit&amp;section=2" title="Edit section: Authentication: OpenSSH key management"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>On <a href="/wiki/Unix-like" title="Unix-like">Unix-like</a> systems, the list of authorized public keys is typically stored in the home directory of the user that is allowed to log in remotely, in the file <code>~/.ssh/authorized_keys</code>.<sup id="cite&#95;ref-4" class="reference"><a href="#cite_note-4"><span class="cite-bracket">&#91;</span>4<span class="cite-bracket">&#93;</span></a></sup> This file is respected by SSH only if it is not writable by anything apart from the owner and root. When the public key is present on the remote end, and the matching private key is present on the local end, typing in the password is no longer required. However, for additional security, the private key itself can be locked with a passphrase.
</p><p>The private key can also be looked for in standard places, and its full path can be specified as a command-line setting (the option <code>-i</code> for ssh). The <a href="/wiki/Ssh-keygen" title="Ssh-keygen">ssh-keygen</a> utility produces the public and private keys, always in pairs.
</p>

<p>SSH is typically used to log into a remote computer's <a href="/wiki/Shell_(computing)" title="Shell (computing)">shell</a> or <a href="/wiki/Command-line_interface" title="Command-line interface">command-line interface</a> (CLI) and to execute commands on a remote server. It also supports mechanisms for <a href="/wiki/Tunneling_protocol" title="Tunneling protocol">tunneling</a>, <a href="/wiki/Port_forwarding" title="Port forwarding">forwarding</a> of <a href="/wiki/TCP_and_UDP_port" class="mw-redirect" title="TCP and UDP port">TCP ports</a> and <a href="/wiki/X11" class="mw-redirect" title="X11">X11</a> connections and it can be used to transfer files using the associated <a href="/wiki/SSH_File_Transfer_Protocol" title="SSH File Transfer Protocol">SSH File Transfer Protocol</a> (SFTP) or <a href="/wiki/Secure_Copy_Protocol" class="mw-redirect" title="Secure Copy Protocol">Secure Copy Protocol</a> (SCP).<sup id="cite&#95;ref-rfc4252&#95;3-2" class="reference"><a href="#cite_note-rfc4252-3"><span class="cite-bracket">&#91;</span>3<span class="cite-bracket">&#93;</span></a></sup>
</p><p>SSH uses the <a href="/wiki/Client%E2%80%93server_model" title="Client–server model">client–server model</a>. An SSH <a href="/wiki/Client_(computing)" title="Client (computing)">client</a> program is typically used for establishing connections to an SSH <a href="/wiki/Daemon_(computing)" title="Daemon (computing)">daemon</a>, such as sshd, accepting remote connections. Both are commonly present on most modern <a href="/wiki/Operating_systems" class="mw-redirect" title="Operating systems">operating systems</a>, including <a href="/wiki/MacOS" title="MacOS">macOS</a>, most distributions of <a href="/wiki/Linux" title="Linux">Linux</a>, <a href="/wiki/OpenBSD" title="OpenBSD">OpenBSD</a>, <a href="/wiki/FreeBSD" title="FreeBSD">FreeBSD</a>, <a href="/wiki/NetBSD" title="NetBSD">NetBSD</a>, <a href="/wiki/Solaris_(operating_system)" class="mw-redirect" title="Solaris (operating system)">Solaris</a> and <a href="/wiki/OpenVMS" title="OpenVMS">OpenVMS</a>. Notably, versions of <a href="/wiki/Windows" class="mw-redirect" title="Windows">Windows</a> prior to Windows 10 version 1709 do not include SSH by default, but <a href="/wiki/Proprietary_software" title="Proprietary software">proprietary</a>, <a href="/wiki/Freeware" title="Freeware">freeware</a> and <a href="/wiki/Open-source_software" title="Open-source software">open source</a> versions of various levels of complexity and completeness did and do exist (see <a href="/wiki/Comparison_of_SSH_clients" title="Comparison of SSH clients">Comparison of SSH clients</a>). In 2018 <a href="/wiki/Microsoft" title="Microsoft">Microsoft</a> began porting the <a href="/wiki/OpenSSH" title="OpenSSH">OpenSSH</a> source code to Windows<sup id="cite&#95;ref-5" class="reference"><a href="#cite_note-5"><span class="cite-bracket">&#91;</span>5<span class="cite-bracket">&#93;</span></a></sup> and in <a href="/wiki/Windows_10_version_1709" class="mw-redirect" title="Windows 10 version 1709">Windows 10 version 1709</a>, an official Win32 port of OpenSSH is now available.
</p><p>File managers for UNIX-like systems (e.g., <a href="/wiki/Konqueror" title="Konqueror">Konqueror</a>) can use the <a href="/wiki/Files_transferred_over_shell_protocol" title="Files transferred over shell protocol">FISH</a> protocol to provide a split-pane GUI with drag-and-drop. The open source Windows program <a href="/wiki/WinSCP" title="WinSCP">WinSCP</a><sup id="cite&#95;ref-6" class="reference"><a href="#cite_note-6"><span class="cite-bracket">&#91;</span>6<span class="cite-bracket">&#93;</span></a></sup> provides similar file management (synchronization, copy, remote delete) capability using <a href="/wiki/PuTTY" title="PuTTY">PuTTY</a> as a back-end. Both WinSCP<sup id="cite&#95;ref-7" class="reference"><a href="#cite_note-7"><span class="cite-bracket">&#91;</span>7<span class="cite-bracket">&#93;</span></a></sup> and PuTTY<sup id="cite&#95;ref-8" class="reference"><a href="#cite_note-8"><span class="cite-bracket">&#91;</span>8<span class="cite-bracket">&#93;</span></a></sup> are available packaged to run directly off a USB drive, without requiring installation on the client machine. Crostini on <a href="/wiki/ChromeOS" title="ChromeOS">ChromeOS</a> comes with OpenSSH by default. Setting up an SSH server in Windows typically involves enabling a feature in the Settings app.
</p><p>SSH is important in <a href="/wiki/Cloud_computing" title="Cloud computing">cloud computing</a> to solve connectivity problems, avoiding the security issues of exposing a cloud-based virtual machine directly on the Internet. An SSH tunnel can provide a secure path over the Internet, through a firewall to a virtual machine.<sup id="cite&#95;ref-9" class="reference"><a href="#cite_note-9"><span class="cite-bracket">&#91;</span>9<span class="cite-bracket">&#93;</span></a></sup>
</p><p>The <a href="/wiki/IANA" class="mw-redirect" title="IANA">IANA</a> has assigned <a href="/wiki/Transmission_Control_Protocol" title="Transmission Control Protocol">TCP</a> <a href="/wiki/Port_(computer_networking)" title="Port (computer networking)">port</a> 22, <a href="/wiki/User_Datagram_Protocol" title="User Datagram Protocol">UDP</a> port 22 and <a href="/wiki/SCTP" class="mw-redirect" title="SCTP">SCTP</a> port 22 for this protocol.<sup id="cite&#95;ref-10" class="reference"><a href="#cite_note-10"><span class="cite-bracket">&#91;</span>10<span class="cite-bracket">&#93;</span></a></sup> IANA had listed the standard TCP port 22 for SSH servers as one of the <a href="/wiki/Well-known_ports" class="mw-redirect" title="Well-known ports">well-known ports</a> as early as 2001.<sup id="cite&#95;ref-11" class="reference"><a href="#cite_note-11"><span class="cite-bracket">&#91;</span>11<span class="cite-bracket">&#93;</span></a></sup> SSH can also be run using <a href="/wiki/SCTP" class="mw-redirect" title="SCTP">SCTP</a> rather than TCP as the connection-oriented transport layer protocol.<sup id="cite&#95;ref-12" class="reference"><a href="#cite_note-12"><span class="cite-bracket">&#91;</span>12<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading2"><h2 id="Historical_development">Historical development</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Secure_Shell&amp;action=edit&amp;section=4" title="Edit section: Historical development"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<div class="mw-heading mw-heading3"><h3 id="Version_1">Version 1</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Secure_Shell&amp;action=edit&amp;section=5" title="Edit section: Version 1"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>In 1995, <b>Tatu Ylönen</b>, a researcher at <a href="/wiki/Helsinki_University_of_Technology" title="Helsinki University of Technology">Helsinki University of Technology</a> in Finland designed the first version of the protocol (now called <b>SSH-1</b>) prompted by a password-<a href="/wiki/Packet_analyzer" title="Packet analyzer">sniffing</a> attack at his <a href="/wiki/University_network" class="mw-redirect" title="University network">university network</a>.<sup id="cite&#95;ref-13" class="reference"><a href="#cite_note-13"><span class="cite-bracket">&#91;</span>13<span class="cite-bracket">&#93;</span></a></sup> The goal of SSH was to replace the earlier <a href="/wiki/Rlogin" class="mw-redirect" title="Rlogin">rlogin</a>, <a href="/wiki/TELNET" class="mw-redirect" title="TELNET">TELNET</a>, <a href="/wiki/FTP" class="mw-redirect" title="FTP">FTP</a><sup id="cite&#95;ref-14" class="reference"><a href="#cite_note-14"><span class="cite-bracket">&#91;</span>14<span class="cite-bracket">&#93;</span></a></sup> and <a href="/wiki/Remote_Shell" title="Remote Shell">rsh</a> protocols, which did not provide strong authentication nor guarantee confidentiality. He chose the port number 22 because it is between <code>telnet</code> (port 23) and <code>ftp</code> (port 21).<sup id="cite&#95;ref-15" class="reference"><a href="#cite_note-15"><span class="cite-bracket">&#91;</span>15<span class="cite-bracket">&#93;</span></a></sup>
</p><p>Ylönen released his implementation as <a href="/wiki/Freeware" title="Freeware">freeware</a> in July 1995, and the tool quickly gained in popularity. Towards the end of 1995, the SSH user base had grown to 20,000 users in fifty countries.<sup id="cite&#95;ref-16" class="reference"><a href="#cite_note-16"><span class="cite-bracket">&#91;</span>16<span class="cite-bracket">&#93;</span></a></sup>
</p><p>In December 1995, Ylönen founded SSH Communications Security to market and develop SSH. The original version of the SSH software used various pieces of <a href="/wiki/Free_software" title="Free software">free software</a>, such as <a href="/wiki/GNU_Multi-Precision_Library" class="mw-redirect" title="GNU Multi-Precision Library">GNU libgmp</a>, but later versions released by SSH Communications Security evolved into increasingly <a href="/wiki/Proprietary_software" title="Proprietary software">proprietary software</a>.
</p><p>It was estimated that by 2000, the number of users had grown to 2 million.<sup id="cite&#95;ref-Nicholas&#95;Rosasco&#95;and&#95;David&#95;Larochelle&#95;17-0" class="reference"><a href="#cite_note-Nicholas_Rosasco_and_David_Larochelle-17"><span class="cite-bracket">&#91;</span>17<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading3"><h3 id="Version_2">Version 2</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Secure_Shell&amp;action=edit&amp;section=6" title="Edit section: Version 2"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>In 2006, after being discussed in a working group named "secsh",<sup id="cite&#95;ref-18" class="reference"><a href="#cite_note-18"><span class="cite-bracket">&#91;</span>18<span class="cite-bracket">&#93;</span></a></sup> a revised version of the SSH protocol, <b>SSH-2</b> was adopted as a standard.<sup id="cite&#95;ref-rfc-ssh&#95;19-0" class="reference"><a href="#cite_note-rfc-ssh-19"><span class="cite-bracket">&#91;</span>19<span class="cite-bracket">&#93;</span></a></sup> This version offers improved security and new features, but is not compatible with SSH-1. For example, it introduces new key-exchange mechanisms like <a href="/wiki/Diffie%E2%80%93Hellman_key_exchange" title="Diffie–Hellman key exchange">Diffie–Hellman key exchange</a>, improved <a href="/wiki/Data_integrity" title="Data integrity">data integrity</a> checking via <a href="/wiki/Message_authentication_code" title="Message authentication code">message authentication codes</a> like <a href="/wiki/MD5" title="MD5">MD5</a> or <a href="/wiki/SHA-1" title="SHA-1">SHA-1</a>, which can be negotiated between client and server. SSH-2 also adds stronger encryption methods like <a href="/wiki/Advanced_Encryption_Standard" title="Advanced Encryption Standard">AES</a> which eventually replaced weaker and compromised ciphers from the previous standard like <a href="/wiki/3DES" class="mw-redirect" title="3DES">3DES</a>.<sup id="cite&#95;ref-20" class="reference"><a href="#cite_note-20"><span class="cite-bracket">&#91;</span>20<span class="cite-bracket">&#93;</span></a></sup><sup id="cite&#95;ref-21" class="reference"><a href="#cite_note-21"><span class="cite-bracket">&#91;</span>21<span class="cite-bracket">&#93;</span></a></sup><sup id="cite&#95;ref-rfc-ssh&#95;19-1" class="reference"><a href="#cite_note-rfc-ssh-19"><span class="cite-bracket">&#91;</span>19<span class="cite-bracket">&#93;</span></a></sup> New features of SSH-2 include the ability to run any number of <a href="/wiki/Unix_shell" title="Unix shell">shell</a> sessions over a single SSH connection.<sup id="cite&#95;ref-22" class="reference"><a href="#cite_note-22"><span class="cite-bracket">&#91;</span>22<span class="cite-bracket">&#93;</span></a></sup> Due to SSH-2's superiority and popularity over SSH-1, some implementations such as libssh (v0.8.0+),<sup id="cite&#95;ref-23" class="reference"><a href="#cite_note-23"><span class="cite-bracket">&#91;</span>23<span class="cite-bracket">&#93;</span></a></sup> <a href="/wiki/Lsh" title="Lsh">Lsh</a><sup id="cite&#95;ref-24" class="reference"><a href="#cite_note-24"><span class="cite-bracket">&#91;</span>24<span class="cite-bracket">&#93;</span></a></sup> and <a href="/wiki/Dropbear_(software)" title="Dropbear (software)">Dropbear</a><sup id="cite&#95;ref-25" class="reference"><a href="#cite_note-25"><span class="cite-bracket">&#91;</span>25<span class="cite-bracket">&#93;</span></a></sup> eventually supported only the SSH-2 protocol.
</p>
<div class="mw-heading mw-heading3"><h3 id="Version_1.99">Version 1.99</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Secure_Shell&amp;action=edit&amp;section=7" title="Edit section: Version 1.99"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>In January 2006, well after version 2.1 was established, <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r1333433106" /><a href="/wiki/RFC_(identifier)" class="mw-redirect" title="RFC (identifier)">RFC</a>&#160;<a rel="nofollow" class="external text" href="https://www.rfc-editor.org/rfc/rfc4253">4253</a> specified that an SSH server supporting 2.0 as well as prior versions should identify its protocol version as 1.99.<sup id="cite&#95;ref-26" class="reference"><a href="#cite_note-26"><span class="cite-bracket">&#91;</span>26<span class="cite-bracket">&#93;</span></a></sup> This version number does not reflect a historical software revision, but a method to identify <a href="/wiki/Backward_compatibility" title="Backward compatibility">backward compatibility</a>.
</p>
<div class="mw-heading mw-heading3"><h3 id="OSSH_and_OpenSSH"><span class="anchor" id="OSSH"></span>OSSH and OpenSSH</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Secure_Shell&amp;action=edit&amp;section=8" title="Edit section: OSSH and OpenSSH"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>In 1999, developers, desiring availability of a free software version, restarted software development from the 1.2.12 release of the original SSH program, which was the last released under an <a href="/wiki/Open_source_license" class="mw-redirect" title="Open source license">open source license</a>.<sup id="cite&#95;ref-27" class="reference"><a href="#cite_note-27"><span class="cite-bracket">&#91;</span>27<span class="cite-bracket">&#93;</span></a></sup> This served as a code base for Björn Grönvall's OSSH software.<sup id="cite&#95;ref-28" class="reference"><a href="#cite_note-28"><span class="cite-bracket">&#91;</span>28<span class="cite-bracket">&#93;</span></a></sup> Shortly thereafter, <a href="/wiki/OpenBSD" title="OpenBSD">OpenBSD</a> developers <a href="/wiki/Fork_(software_development)" title="Fork (software development)">forked</a> Grönvall's code and created <a href="/wiki/OpenSSH" title="OpenSSH">OpenSSH</a>, which shipped with Release 2.6 of OpenBSD. From this version, a "portability" branch was formed to port OpenSSH to other operating systems.<sup id="cite&#95;ref-29" class="reference"><a href="#cite_note-29"><span class="cite-bracket">&#91;</span>29<span class="cite-bracket">&#93;</span></a></sup>
</p><p>As of 2005<sup class="plainlinks noexcerpt noprint asof-tag update" style="display:none;"><a class="external text" href="https://en.wikipedia.org/w/index.php?title=Secure_Shell&amp;action=edit">&#91;update&#93;</a></sup>, <a href="/wiki/OpenSSH" title="OpenSSH">OpenSSH</a> was the single most popular SSH implementation, being the default version in a large number of operating system distributions. OSSH, meanwhile, has become obsolete.<sup id="cite&#95;ref-30" class="reference"><a href="#cite_note-30"><span class="cite-bracket">&#91;</span>30<span class="cite-bracket">&#93;</span></a></sup> OpenSSH continues to be maintained and supports the SSH-2 protocol, having expunged SSH-1 support from the codebase in the OpenSSH 7.6 release.
</p>
<div class="mw-heading mw-heading3"><h3 id="Future">Future</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Secure_Shell&amp;action=edit&amp;section=9" title="Edit section: Future"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>In 2023, an alternative to traditional SSH was proposed under the name <b>SSH3</b><sup id="cite&#95;ref-31" class="reference"><a href="#cite_note-31"><span class="cite-bracket">&#91;</span>31<span class="cite-bracket">&#93;</span></a></sup><sup id="cite&#95;ref-32" class="reference"><a href="#cite_note-32"><span class="cite-bracket">&#91;</span>32<span class="cite-bracket">&#93;</span></a></sup><sup id="cite&#95;ref-33" class="reference"><a href="#cite_note-33"><span class="cite-bracket">&#91;</span>33<span class="cite-bracket">&#93;</span></a></sup> by PhD student François Michel and Professor Olivier Bonaventure and its code has been made open source.<sup id="cite&#95;ref-34" class="reference"><a href="#cite_note-34"><span class="cite-bracket">&#91;</span>34<span class="cite-bracket">&#93;</span></a></sup> This new version implements the original SSH Connection Protocol but operates on top of <a href="/wiki/HTTP/3" title="HTTP/3">HTTP/3</a>, which runs on <a href="/wiki/QUIC" title="QUIC">QUIC</a>. It offers multiple features such as:
</p>
<ul><li>Faster session establishment, reducing the number of <a href="/wiki/Round-trip_delay" title="Round-trip delay">Round-trip delays</a> from 5-7 to 3.</li>
<li>High security: while SSHv2 relies on its own protocols, SSH3 leverages <a href="/wiki/TLS_1.3" class="mw-redirect" title="TLS 1.3">TLS 1.3</a>, <a href="/wiki/QUIC" title="QUIC">QUIC</a>, and <a href="/wiki/HTTP" title="HTTP">HTTP</a>.</li>
<li>UDP port forwarding</li>
<li>X.509 certificates</li>
<li>OpenID Connect</li></ul>
<p>However, the name SSH3 is under discussion, and the project aims to rename itself to a more suitable name.<sup id="cite&#95;ref-35" class="reference"><a href="#cite_note-35"><span class="cite-bracket">&#91;</span>35<span class="cite-bracket">&#93;</span></a></sup> The discussion stems from the fact that this new implementation significantly revises the SSH protocol, suggesting it should not be called SSH3.
</p>
<div class="mw-heading mw-heading2"><h2 id="Uses">Uses</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Secure_Shell&amp;action=edit&amp;section=10" title="Edit section: Uses"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<figure class="mw-halign-right" typeof="mw:File/Thumb"><a href="/wiki/File:X11_ssh_tunnelling.png" class="mw-file-description"><img src="//upload.wikimedia.org/wikipedia/commons/thumb/0/03/X11_ssh_tunnelling.png/250px-X11_ssh_tunnelling.png" decoding="async" width="250" height="166" class="mw-file-element" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/0/03/X11_ssh_tunnelling.png/500px-X11_ssh_tunnelling.png 1.5x" data-file-width="583" data-file-height="386" /></a><figcaption>Example of tunneling an <a href="/wiki/X11" class="mw-redirect" title="X11">X11</a> application over SSH: the user 'josh' has "SSHed" from the local machine 'foofighter' to the remote machine 'tengwar' to run <a href="/wiki/Xeyes" class="mw-redirect" title="Xeyes">xeyes</a>.</figcaption></figure>
<figure class="mw-halign-right" typeof="mw:File/Thumb"><a href="/wiki/File:OpenWrtPuTTY.png" class="mw-file-description"><img src="//upload.wikimedia.org/wikipedia/commons/thumb/1/1e/OpenWrtPuTTY.png/250px-OpenWrtPuTTY.png" decoding="async" width="250" height="125" class="mw-file-element" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/1/1e/OpenWrtPuTTY.png/500px-OpenWrtPuTTY.png 1.5x" data-file-width="826" data-file-height="413" /></a><figcaption>Logging into <a href="/wiki/OpenWrt" title="OpenWrt">OpenWrt</a> via SSH using <a href="/wiki/PuTTY" title="PuTTY">PuTTY</a> running on <a href="/wiki/Windows" class="mw-redirect" title="Windows">Windows</a>.</figcaption></figure>
<p>SSH is a protocol that can be used for many applications across many platforms including most <a href="/wiki/Unix" title="Unix">Unix</a> variants (<a href="/wiki/Linux" title="Linux">Linux</a>, the <a href="/wiki/BSD" class="mw-redirect" title="BSD">BSDs</a> including <a href="/wiki/Apple_Inc" class="mw-redirect" title="Apple Inc">Apple</a>'s <a href="/wiki/MacOS" title="MacOS">macOS</a>, and <a href="/wiki/Solaris_(operating_system)" class="mw-redirect" title="Solaris (operating system)">Solaris</a>), as well as <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Microsoft Windows</a>. Some of the applications below may require features that are only available or compatible with specific SSH clients or servers. For example, using the SSH protocol to implement a <a href="/wiki/VPN" class="mw-redirect" title="VPN">VPN</a> is possible, but presently only with the <a href="/wiki/OpenSSH" title="OpenSSH">OpenSSH</a> server and client implementation.
</p>
<ul><li>For login to a shell on a remote host (replacing <a href="/wiki/Telnet" title="Telnet">Telnet</a> and <a href="/wiki/Rlogin" class="mw-redirect" title="Rlogin">rlogin</a>)</li>
<li>For executing a single command on a remote host (replacing <a href="/wiki/Remote_shell" class="mw-redirect" title="Remote shell">rsh</a>)</li>
<li>For setting up automatic (passwordless) login to a remote server (for example, using <a href="/wiki/OpenSSH" title="OpenSSH">OpenSSH</a><sup id="cite&#95;ref-36" class="reference"><a href="#cite_note-36"><span class="cite-bracket">&#91;</span>36<span class="cite-bracket">&#93;</span></a></sup>)</li>
<li>In combination with <a href="/wiki/Rsync" title="Rsync">rsync</a> to back up, copy and mirror files efficiently and securely</li>
<li>For <a href="/wiki/Port_forwarding" title="Port forwarding">forwarding</a> a port</li>
<li>For <a href="/wiki/Tunneling_protocol" title="Tunneling protocol">tunneling</a> (not to be confused with a <a href="/wiki/VPN" class="mw-redirect" title="VPN">VPN</a>, which <a href="/wiki/VPN#Routing" class="mw-redirect" title="VPN">routes</a> packets between different networks, or <a href="/wiki/VPN#OSI_Layer_1_services" class="mw-redirect" title="VPN">bridges</a> two <a href="/wiki/Broadcast_domain" title="Broadcast domain">broadcast domains</a> into one).</li>
<li>For use as a full-fledged encrypted VPN. Note that only <a href="/wiki/OpenSSH" title="OpenSSH">OpenSSH</a> server and client support this feature.</li>
<li>For forwarding <a href="/wiki/X_Window_System" title="X Window System">X</a> from a remote <a href="/wiki/Host_(network)" title="Host (network)">host</a> (possible through multiple intermediate hosts)</li>
<li>For browsing the web through an encrypted proxy connection with SSH clients that support the <a href="/wiki/SOCKS" title="SOCKS">SOCKS protocol</a>.</li>
<li>For securely mounting a directory on a remote server as a <a href="/wiki/File_system" title="File system">file system</a> on a local computer using <a href="/wiki/SSHFS" title="SSHFS">SSHFS</a>.</li>
<li>For automated remote monitoring and management of servers through one or more of the mechanisms discussed above.</li>
<li>For development on a mobile or embedded device that supports SSH.</li>
<li>For securing file transfer protocols.</li></ul>
<div class="mw-heading mw-heading3"><h3 id="File_transfer_protocols">File transfer protocols</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Secure_Shell&amp;action=edit&amp;section=11" title="Edit section: File transfer protocols"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>The Secure Shell protocols are used in several file transfer mechanisms.
</p>
<ul><li><a href="/wiki/Secure_copy" class="mw-redirect" title="Secure copy">Secure copy</a> (SCP), which evolved from <a href="/wiki/Rcp_(Unix)" class="mw-redirect" title="Rcp (Unix)">RCP</a> protocol over SSH</li>
<li><a href="/wiki/Rsync" title="Rsync">rsync</a>, intended to be more efficient than SCP. Generally runs over an SSH connection.</li>
<li><a href="/wiki/SSH_File_Transfer_Protocol" title="SSH File Transfer Protocol">SSH File Transfer Protocol</a> (SFTP), a secure alternative to <a href="/wiki/FTP" class="mw-redirect" title="FTP">FTP</a> (not to be confused with <a href="/wiki/FTP_over_SSH" class="mw-redirect" title="FTP over SSH">FTP over SSH</a> or <a href="/wiki/FTPS" title="FTPS">FTPS</a>)</li>
<li><a href="/wiki/Files_transferred_over_shell_protocol" title="Files transferred over shell protocol">Files transferred over shell protocol</a> (FISH), released in 1998, which evolved from <a href="/wiki/Unix_shell" title="Unix shell">Unix shell</a> commands over SSH</li>
<li><a href="/wiki/Fast_and_Secure_Protocol" title="Fast and Secure Protocol">Fast and Secure Protocol</a> (FASP), aka <i>Aspera</i>, uses SSH for control and UDP ports for data transfer.</li></ul>
<div class="mw-heading mw-heading2"><h2 id="Architecture"><span class="anchor" id="ARCHITECTURE"></span>Architecture</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Secure_Shell&amp;action=edit&amp;section=12" title="Edit section: Architecture"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<figure class="mw-halign-right" typeof="mw:File/Thumb"><a href="/wiki/File:Ssh_binary_packet_alt.svg" class="mw-file-description"><img src="//upload.wikimedia.org/wikipedia/commons/thumb/0/0f/Ssh_binary_packet_alt.svg/250px-Ssh_binary_packet_alt.svg.png" decoding="async" width="250" height="193" class="mw-file-element" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/0/0f/Ssh_binary_packet_alt.svg/500px-Ssh_binary_packet_alt.svg.png 1.5x" data-file-width="3300" data-file-height="2550" /></a><figcaption>Diagram of the SSH-2 binary packet.</figcaption></figure>
<p>The SSH protocol has a layered architecture with three separate components:
</p>
<ul><li>The <i>transport layer</i> (<link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r1333433106" /><a href="/wiki/RFC_(identifier)" class="mw-redirect" title="RFC (identifier)">RFC</a>&#160;<a rel="nofollow" class="external text" href="https://www.rfc-editor.org/rfc/rfc4253">4253</a>) typically uses the <a href="/wiki/Transmission_Control_Protocol" title="Transmission Control Protocol">Transmission Control Protocol</a> (TCP) of <a href="/wiki/TCP/IP" class="mw-redirect" title="TCP/IP">TCP/IP</a>, reserving <a href="/wiki/Port_number#Common_port_numbers" class="mw-redirect" title="Port number">port number</a> 22 as a server listening port. This layer handles initial key exchange as well as server authentication, and sets up encryption, compression, and integrity verification. It exposes to the upper layer an interface for sending and receiving plaintext packets with a size of up to 32,768 bytes each, but more can be allowed by each implementation. The transport layer also arranges for key re-exchange, usually after 1 GB of data has been transferred or after one hour has passed, whichever occurs first.</li>
<li>The <i>user authentication layer</i> (<link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r1333433106" /><a href="/wiki/RFC_(identifier)" class="mw-redirect" title="RFC (identifier)">RFC</a>&#160;<a rel="nofollow" class="external text" href="https://www.rfc-editor.org/rfc/rfc4252">4252</a>) handles client authentication, and provides a suite of authentication algorithms. Authentication is <i>client-driven</i>: when one is prompted for a password, it may be the SSH client prompting, not the server. The server merely responds to the client's authentication requests. Widely used user-authentication methods include the following:
<ul><li><i>password</i>: a method for straightforward password authentication, including a facility allowing a password to be changed. Not all programs implement this method.</li>
<li><i>publickey</i>: a method for <a href="/wiki/Public-key_cryptography" title="Public-key cryptography">public-key-based authentication</a>, usually supporting at least <a href="/wiki/Digital_Signature_Algorithm" title="Digital Signature Algorithm">DSA</a>, <a href="/wiki/ECDSA" class="mw-redirect" title="ECDSA">ECDSA</a> or <a href="/wiki/RSA_(algorithm)" class="mw-redirect" title="RSA (algorithm)">RSA</a> keypairs, with other implementations also supporting <a href="/wiki/X.509" title="X.509">X.509</a> certificates.</li>
<li><i>keyboard-interactive</i> (<link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r1333433106" /><a href="/wiki/RFC_(identifier)" class="mw-redirect" title="RFC (identifier)">RFC</a>&#160;<a rel="nofollow" class="external text" href="https://www.rfc-editor.org/rfc/rfc4256">4256</a>): a versatile method where the server sends one or more prompts to enter information and the client displays them and sends back responses keyed in by the user. Used to provide <a href="/wiki/One-time_password" title="One-time password">one-time password</a> authentication such as <a href="/wiki/S/Key" class="mw-redirect" title="S/Key">S/Key</a> or <a href="/wiki/SecurID" class="mw-redirect" title="SecurID">SecurID</a>. Used by some OpenSSH configurations when <a href="/wiki/Pluggable_authentication_modules" class="mw-redirect" title="Pluggable authentication modules">PAM</a> is the underlying host-authentication provider to effectively provide password authentication, sometimes leading to an inability to log in with a client that supports just the plain <i>password</i> authentication method.</li>
<li><a href="/wiki/GSSAPI" class="mw-redirect" title="GSSAPI">GSSAPI</a> authentication methods which provide an extensible scheme to perform SSH authentication using external mechanisms such as <a href="/wiki/Kerberos_5" class="mw-redirect" title="Kerberos 5">Kerberos 5</a> or <a href="/wiki/NTLM" title="NTLM">NTLM</a>, providing <a href="/wiki/Single_sign-on" title="Single sign-on">single sign-on</a> capability to SSH sessions. These methods are usually implemented by commercial SSH implementations for use in organizations, though OpenSSH does have a working GSSAPI implementation.</li></ul></li>
<li>The <i>connection layer</i> (<link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r1333433106" /><a href="/wiki/RFC_(identifier)" class="mw-redirect" title="RFC (identifier)">RFC</a>&#160;<a rel="nofollow" class="external text" href="https://www.rfc-editor.org/rfc/rfc4254">4254</a>) defines the concept of channels, channel requests, and global requests, which define the SSH services provided. A single SSH connection can be multiplexed into multiple logical channels simultaneously, each transferring data bidirectionally. Channel requests are used to relay out-of-band channel-specific data, such as the changed size of a terminal window or the exit code of a server-side process. Additionally, each channel performs its own flow control using the receive window size. The SSH client requests a server-side port to be forwarded using a global request. Standard channel types include:
<ul><li><i>shell</i> for terminal shells, SFTP and exec requests (including SCP transfers)</li>
<li><i>direct-tcpip</i> for client-to-server forwarded connections</li>
<li><i>forwarded-tcpip</i> for server-to-client forwarded connections</li></ul></li>
<li>The <a href="/wiki/SSHFP" class="mw-redirect" title="SSHFP">SSHFP</a> DNS record (RFC 4255) provides the public host key fingerprints in order to aid in verifying the authenticity of the host.</li></ul>
<p>This open architecture provides considerable flexibility, allowing the use of SSH for a variety of purposes beyond a secure shell. The functionality of the transport layer alone is comparable to <a href="/wiki/Transport_Layer_Security" title="Transport Layer Security">Transport Layer Security</a> (TLS); the user-authentication layer is highly extensible with custom authentication methods; and the connection layer provides the ability to multiplex many secondary sessions into a single SSH connection, a feature comparable to <a href="/wiki/BEEP" title="BEEP">BEEP</a> and not available in TLS.
</p>
<div class="mw-heading mw-heading2"><h2 id="Algorithms">Algorithms</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Secure_Shell&amp;action=edit&amp;section=13" title="Edit section: Algorithms"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<ul><li><a href="/wiki/EdDSA" title="EdDSA">EdDSA</a>,<sup id="cite&#95;ref-37" class="reference"><a href="#cite_note-37"><span class="cite-bracket">&#91;</span>37<span class="cite-bracket">&#93;</span></a></sup> <a href="/wiki/ECDSA" class="mw-redirect" title="ECDSA">ECDSA</a>, <a href="/wiki/RSA_(cryptosystem)" class="mw-redirect" title="RSA (cryptosystem)">RSA</a> and <a href="/wiki/Digital_Signature_Algorithm" title="Digital Signature Algorithm">DSA</a> for <a href="/wiki/Public-key_cryptography" title="Public-key cryptography">public-key cryptography</a>.<sup id="cite&#95;ref-&#82;FC5656&#95;38-0" class="reference"><a href="#cite_note-RFC5656-38"><span class="cite-bracket">&#91;</span>38<span class="cite-bracket">&#93;</span></a></sup></li>
<li><a href="/wiki/ECDH" class="mw-redirect" title="ECDH">ECDH</a> and <a href="/wiki/Diffie%E2%80%93Hellman" class="mw-redirect" title="Diffie–Hellman">Diffie–Hellman</a> for <a href="/wiki/Key_exchange" title="Key exchange">key exchange</a>.<sup id="cite&#95;ref-&#82;FC5656&#95;38-1" class="reference"><a href="#cite_note-RFC5656-38"><span class="cite-bracket">&#91;</span>38<span class="cite-bracket">&#93;</span></a></sup></li>
<li><a href="/wiki/HMAC" title="HMAC">HMAC</a>, <a href="/wiki/AEAD" class="mw-redirect" title="AEAD">AEAD</a> and <a href="/wiki/UMAC_(cryptography)" title="UMAC (cryptography)">UMAC</a> for <a href="/wiki/Message_authentication_code" title="Message authentication code">MAC</a>.<sup id="cite&#95;ref-39" class="reference"><a href="#cite_note-39"><span class="cite-bracket">&#91;</span>39<span class="cite-bracket">&#93;</span></a></sup></li>
<li><a href="/wiki/Advanced_Encryption_Standard" title="Advanced Encryption Standard">AES</a> (and deprecated <a href="/wiki/RC4" title="RC4">RC4</a>, <a href="/wiki/3DES" class="mw-redirect" title="3DES">3DES</a>, <a href="/wiki/Data_Encryption_Standard" title="Data Encryption Standard">DES</a><sup id="cite&#95;ref-40" class="reference"><a href="#cite_note-40"><span class="cite-bracket">&#91;</span>40<span class="cite-bracket">&#93;</span></a></sup>) for <a href="/wiki/Symmetric_encryption" class="mw-redirect" title="Symmetric encryption">symmetric encryption</a>.</li>
<li><a href="/wiki/AES-GCM" class="mw-redirect" title="AES-GCM">AES-GCM</a><sup id="cite&#95;ref-41" class="reference"><a href="#cite_note-41"><span class="cite-bracket">&#91;</span>41<span class="cite-bracket">&#93;</span></a></sup> and <a href="/wiki/ChaCha20-Poly1305" title="ChaCha20-Poly1305">ChaCha20-Poly1305</a> for <a href="/wiki/AEAD" class="mw-redirect" title="AEAD">AEAD</a> encryption.</li>
<li><a href="/wiki/Secure_Hash_Algorithm" class="mw-redirect" title="Secure Hash Algorithm">SHA</a> (and deprecated <a href="/wiki/MD5" title="MD5">MD5</a>) for <a href="/wiki/Key_fingerprint" class="mw-redirect" title="Key fingerprint">key fingerprint</a>.</li></ul>
<div class="mw-heading mw-heading2"><h2 id="Vulnerabilities">Vulnerabilities</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Secure_Shell&amp;action=edit&amp;section=14" title="Edit section: Vulnerabilities"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<div class="mw-heading mw-heading3"><h3 id="SSH-1">SSH-1</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Secure_Shell&amp;action=edit&amp;section=15" title="Edit section: SSH-1"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>In 1998, a vulnerability was described in SSH 1.5 which allowed the unauthorized insertion of content into an encrypted SSH stream due to insufficient data integrity protection from <a href="/wiki/CRC-32" class="mw-redirect" title="CRC-32">CRC-32</a> used in this version of the protocol.<sup id="cite&#95;ref-42" class="reference"><a href="#cite_note-42"><span class="cite-bracket">&#91;</span>42<span class="cite-bracket">&#93;</span></a></sup><sup id="cite&#95;ref-43" class="reference"><a href="#cite_note-43"><span class="cite-bracket">&#91;</span>43<span class="cite-bracket">&#93;</span></a></sup> A fix known as SSH Compensation Attack Detector<sup id="cite&#95;ref-44" class="reference"><a href="#cite_note-44"><span class="cite-bracket">&#91;</span>44<span class="cite-bracket">&#93;</span></a></sup> was introduced into most implementations. Many of these updated implementations contained a new <a href="/wiki/Integer_overflow" title="Integer overflow">integer overflow</a> vulnerability<sup id="cite&#95;ref-45" class="reference"><a href="#cite_note-45"><span class="cite-bracket">&#91;</span>45<span class="cite-bracket">&#93;</span></a></sup> that allowed attackers to execute arbitrary code with the privileges of the SSH daemon, typically root.
</p><p>In January 2001 a vulnerability was discovered that allows attackers to modify the last block of an <a href="/wiki/International_Data_Encryption_Algorithm" title="International Data Encryption Algorithm">IDEA</a>-encrypted session.<sup id="cite&#95;ref-46" class="reference"><a href="#cite_note-46"><span class="cite-bracket">&#91;</span>46<span class="cite-bracket">&#93;</span></a></sup> The same month, another vulnerability was discovered that allowed a malicious server to forward a client authentication to another server.<sup id="cite&#95;ref-cert1&#95;47-0" class="reference"><a href="#cite_note-cert1-47"><span class="cite-bracket">&#91;</span>47<span class="cite-bracket">&#93;</span></a></sup>
</p><p>Since SSH-1 has inherent design flaws that make it vulnerable, it is now generally considered obsolete and should be avoided by explicitly disabling fallback to SSH-1. Most modern servers and clients support SSH-2.<sup id="cite&#95;ref-cert1&#95;47-1" class="reference"><a href="#cite_note-cert1-47"><span class="cite-bracket">&#91;</span>47<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading3"><h3 id="CBC_plaintext_recovery">CBC plaintext recovery</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Secure_Shell&amp;action=edit&amp;section=16" title="Edit section: CBC plaintext recovery"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>In November 2008, a theoretical vulnerability was discovered for all versions of SSH which allowed recovery of up to 32 bits of plaintext from a block of ciphertext that was encrypted using what was then the standard default encryption mode, <a href="/wiki/Block_cipher_modes_of_operation#Cipher-block_chaining_(CBC)" class="mw-redirect" title="Block cipher modes of operation">CBC</a>.<sup id="cite&#95;ref-SSH&#95;CBC&#95;vulnerability&#95;48-0" class="reference"><a href="#cite_note-SSH_CBC_vulnerability-48"><span class="cite-bracket">&#91;</span>48<span class="cite-bracket">&#93;</span></a></sup> The most straightforward solution is to use <a href="/wiki/Block_cipher_modes_of_operation#Counter_(CTR)" class="mw-redirect" title="Block cipher modes of operation">CTR</a>, counter mode, instead of CBC mode, since this renders SSH resistant to the attack.<sup id="cite&#95;ref-SSH&#95;CBC&#95;vulnerability&#95;48-1" class="reference"><a href="#cite_note-SSH_CBC_vulnerability-48"><span class="cite-bracket">&#91;</span>48<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading3"><h3 id="Suspected_decryption_by_NSA">Suspected decryption by NSA</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Secure_Shell&amp;action=edit&amp;section=17" title="Edit section: Suspected decryption by NSA"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>On December 28, 2014 <i><a href="/wiki/Der_Spiegel" title="Der Spiegel">Der Spiegel</a></i> published classified information<sup id="cite&#95;ref-Spiegel2014&#95;49-0" class="reference"><a href="#cite_note-Spiegel2014-49"><span class="cite-bracket">&#91;</span>49<span class="cite-bracket">&#93;</span></a></sup> leaked by whistleblower <a href="/wiki/Edward_Snowden" title="Edward Snowden">Edward Snowden</a> which suggests that the <a href="/wiki/National_Security_Agency" title="National Security Agency">National Security Agency</a> may be able to decrypt some SSH traffic. The technical details associated with such a process were not disclosed. A 2017 analysis of the <a href="/wiki/CIA" class="mw-redirect" title="CIA">CIA</a> hacking tools <i>BothanSpy</i> and <i>Gyrfalcon</i> suggested that the SSH protocol was not compromised.<sup id="cite&#95;ref-50" class="reference"><a href="#cite_note-50"><span class="cite-bracket">&#91;</span>50<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading3"><h3 id="Terrapin_attack">Terrapin attack</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Secure_Shell&amp;action=edit&amp;section=18" title="Edit section: Terrapin attack"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r1320445320" /><div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Terrapin_attack" title="Terrapin attack">Terrapin attack</a></div>
<p>A novel man-in-the-middle attack against most current ssh implementations was discovered in 2023. It was named the <a href="/wiki/Terrapin_attack" title="Terrapin attack">Terrapin attack</a> by its discoverers.<sup id="cite&#95;ref-51" class="reference"><a href="#cite_note-51"><span class="cite-bracket">&#91;</span>51<span class="cite-bracket">&#93;</span></a></sup><sup id="cite&#95;ref-52" class="reference"><a href="#cite_note-52"><span class="cite-bracket">&#91;</span>52<span class="cite-bracket">&#93;</span></a></sup> However, the risk is mitigated by the requirement to intercept a genuine ssh session, and that the attack is restricted in its scope, fortuitously resulting mostly in failed connections.<sup id="cite&#95;ref-El&#95;Reg&#95;53-0" class="reference"><a href="#cite_note-El_Reg-53"><span class="cite-bracket">&#91;</span>53<span class="cite-bracket">&#93;</span></a></sup><sup id="cite&#95;ref-:1&#95;54-0" class="reference"><a href="#cite_note-:1-54"><span class="cite-bracket">&#91;</span>54<span class="cite-bracket">&#93;</span></a></sup> The ssh developers have stated that the major impact of the attack is to degrade the <a href="/wiki/Keystroke_dynamics" title="Keystroke dynamics">keystroke timing</a> obfuscation features of ssh.<sup id="cite&#95;ref-:1&#95;54-1" class="reference"><a href="#cite_note-:1-54"><span class="cite-bracket">&#91;</span>54<span class="cite-bracket">&#93;</span></a></sup> The vulnerability was fixed in OpenSSH 9.6, but requires both client and server to be upgraded for the fix to be fully effective.
</p>
