<p>The <b>Internet Protocol</b> (<b>IP</b>) is the <a href="/wiki/Network_layer" title="Network layer">network layer</a> <a href="/wiki/Communications_protocol" class="mw-redirect" title="Communications protocol">communications protocol</a> in the <a href="/wiki/Internet_protocol_suite" title="Internet protocol suite">Internet protocol suite</a> for relaying <a href="/wiki/Datagram" title="Datagram">datagrams</a> across network boundaries. Its <a href="/wiki/Routing" title="Routing">routing</a> function enables <a href="/wiki/Internetworking" title="Internetworking">internetworking</a>, and essentially establishes the <a href="/wiki/Internet" title="Internet">Internet</a>.
</p><p>IP has the task of delivering <a href="/wiki/Packet_(information_technology)" class="mw-redirect" title="Packet (information technology)">packets</a> from the source <a href="/wiki/Host_(network)" title="Host (network)">host</a> to the destination host solely based on the <a href="/wiki/IP_address" title="IP address">IP addresses</a> in the packet <a href="/wiki/Header_(computing)" title="Header (computing)">headers</a>. For this purpose, IP defines packet structures that <a href="/wiki/Encapsulation_(networking)" title="Encapsulation (networking)">encapsulate</a> the data to be delivered. It also defines addressing methods that are used to label the datagram with source and destination information.
IP was the <a href="/wiki/Connectionless" class="mw-redirect" title="Connectionless">connectionless</a> datagram service in the original <i><a href="/wiki/Transmission_Control_Program" class="mw-redirect" title="Transmission Control Program">Transmission Control Program</a></i> introduced by <a href="/wiki/Vint_Cerf" title="Vint Cerf">Vint Cerf</a> and <a href="/wiki/Bob_Kahn" class="mw-redirect" title="Bob Kahn">Bob Kahn</a> in 1974, which was complemented by a <a href="/wiki/Connection-oriented" class="mw-redirect" title="Connection-oriented">connection-oriented</a> service that became the basis for the <a href="/wiki/Transmission_Control_Protocol" title="Transmission Control Protocol">Transmission Control Protocol</a> (TCP). The Internet protocol suite is therefore often referred to as <i>TCP/IP</i>.
</p><p>The first major version of IP, <a href="/wiki/Internet_Protocol_version_4" class="mw-redirect" title="Internet Protocol version 4">Internet Protocol version 4</a> (IPv4), is the dominant protocol of the Internet. Its successor is <a href="/wiki/Internet_Protocol_version_6" class="mw-redirect" title="Internet Protocol version 6">Internet Protocol version 6</a> (IPv6), which has been in increasing <a href="/wiki/IPv6_deployment" title="IPv6 deployment">deployment</a> on the public Internet since around 2006.<sup id="cite&#95;ref-1" class="reference"><a href="#cite_note-1"><span class="cite-bracket">&#91;</span>1<span class="cite-bracket">&#93;</span></a></sup>
</p>
<meta property="mw:PageProp/toc" />
<div class="mw-heading mw-heading2"><h2 id="Function">Function</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Internet_Protocol&amp;action=edit&amp;section=1" title="Edit section: Function"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<figure typeof="mw:File/Thumb"><a href="/wiki/File:UDP_encapsulation.svg" class="mw-file-description"><img src="//upload.wikimedia.org/wikipedia/commons/thumb/3/3b/UDP_encapsulation.svg/330px-UDP_encapsulation.svg.png" decoding="async" width="260" height="163" class="mw-file-element" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/3/3b/UDP_encapsulation.svg/500px-UDP_encapsulation.svg.png 1.5x, //upload.wikimedia.org/wikipedia/commons/thumb/3/3b/UDP_encapsulation.svg/960px-UDP_encapsulation.svg.png 2x" data-file-width="800" data-file-height="500" /></a><figcaption>Encapsulation of application data carried by <a href="/wiki/User_Datagram_Protocol" title="User Datagram Protocol">UDP</a> to a link protocol frame</figcaption></figure>
<p>The Internet Protocol is responsible for addressing <a href="/wiki/Host_interface" class="mw-redirect" title="Host interface">host interfaces</a>, encapsulating data into datagrams (including <a href="/wiki/IP_fragmentation" title="IP fragmentation">fragmentation and reassembly</a>) and routing datagrams from a source host interface to a destination host interface across one or more IP networks.<b><sup id="cite&#95;ref-2" class="reference"><a href="#cite_note-2"><span class="cite-bracket">&#91;</span>2<span class="cite-bracket">&#93;</span></a></sup></b> For these purposes, the Internet Protocol defines the format of packets and provides an addressing system.
</p><p>Each datagram has two components: a <a href="/wiki/Header_(computing)" title="Header (computing)">header</a> and a <a href="/wiki/Payload_(computing)" title="Payload (computing)">payload</a>. The <a href="/wiki/IP_header" title="IP header">IP header</a> includes a source IP address, a destination IP address, and other metadata needed to route and deliver the datagram. The payload is the data that is transported. This method of nesting the data payload in a packet with a header is called encapsulation.
</p><p>IP addressing entails the assignment of IP addresses and associated parameters to host interfaces. The address space is divided into <a href="/wiki/Subnet" title="Subnet">subnets</a>, involving the designation of network prefixes. IP routing is performed by all hosts, as well as <a href="/wiki/Router_(computing)" title="Router (computing)">routers</a>, whose main function is to transport packets across network boundaries. Routers communicate with one another via specially designed <a href="/wiki/Routing_protocol" title="Routing protocol">routing protocols</a>, either <a href="/wiki/Interior_gateway_protocol" title="Interior gateway protocol">interior gateway protocols</a> or <a href="/wiki/Exterior_gateway_protocol" title="Exterior gateway protocol">exterior gateway protocols</a>, as needed for the topology of the network.<sup id="cite&#95;ref-3" class="reference"><a href="#cite_note-3"><span class="cite-bracket">&#91;</span>3<span class="cite-bracket">&#93;</span></a></sup>
</p>

<p>There are four principal addressing methods in the Internet Protocol:
</p>
<ul><li><a href="/wiki/Unicast" title="Unicast">Unicast</a> delivers a message to a single specific node using a <i>one-to-one</i> association between a sender and destination: each destination address uniquely identifies a single receiver endpoint.</li>
<li><a href="/wiki/Broadcasting_(networking)" title="Broadcasting (networking)">Broadcast</a> delivers a message to all nodes in the network using a <i>one-to-all</i> association; a single <a href="/wiki/Datagram" title="Datagram">datagram</a> (or <a href="/wiki/Packet_(information_technology)" class="mw-redirect" title="Packet (information technology)">packet</a>) from one sender is routed to all of the possibly multiple endpoints associated with the <a href="/wiki/Broadcast_address" title="Broadcast address">broadcast address</a>. The network automatically replicates datagrams as needed to reach all the recipients within the scope of the broadcast, which is generally an entire network <a href="/wiki/Subnetwork" class="mw-redirect" title="Subnetwork">subnet</a>.</li>
<li><a href="/wiki/Multicast" title="Multicast">Multicast</a> delivers a message to a group of nodes that have expressed interest in receiving the message using a <i>one-to-many-of-many</i> or <i>many-to-many-of-many</i> association; datagrams are routed simultaneously in a single transmission to many recipients. Multicast differs from broadcast in that the destination address designates a subset, not necessarily all, of the accessible nodes.</li>
<li><a href="/wiki/Anycast" title="Anycast">Anycast</a> delivers a message to any one out of a group of nodes, typically the one nearest to the source using a <i>one-to-one-of-many</i><sup id="cite&#95;ref-4" class="reference"><a href="#cite_note-4"><span class="cite-bracket">&#91;</span>4<span class="cite-bracket">&#93;</span></a></sup> association where datagrams are routed to any single member of a group of potential receivers that are all identified by the same destination address.  The routing algorithm selects the single receiver from the group based on which is the nearest according to some distance or cost measure.</li></ul>
<div class="mw-heading mw-heading2"><h2 id="Version_history">Version history</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Internet_Protocol&amp;action=edit&amp;section=3" title="Edit section: Version history"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>

<p>In May 1974, the <a href="/wiki/Institute_of_Electrical_and_Electronics_Engineers" title="Institute of Electrical and Electronics Engineers">Institute of Electrical and Electronics Engineers</a> (IEEE) published a paper entitled "A Protocol for Packet Network Intercommunication".<sup id="cite&#95;ref-5" class="reference"><a href="#cite_note-5"><span class="cite-bracket">&#91;</span>5<span class="cite-bracket">&#93;</span></a></sup> The paper's authors, <a href="/wiki/Vint_Cerf" title="Vint Cerf">Vint Cerf</a> and <a href="/wiki/Bob_Kahn" class="mw-redirect" title="Bob Kahn">Bob Kahn</a>, described an <a href="/wiki/Internetworking" title="Internetworking">internetworking</a> protocol for sharing resources using <a href="/wiki/Packet_switching" title="Packet switching">packet switching</a> among <a href="/wiki/Network_node" class="mw-redirect" title="Network node">network nodes</a>. A central control component of this model was the Transmission Control Program that incorporated both connection-oriented links and datagram services between hosts. The monolithic Transmission Control Program was later divided into a modular architecture consisting of the <a href="/wiki/Transmission_Control_Protocol" title="Transmission Control Protocol">Transmission Control Protocol</a> and <a href="/wiki/User_Datagram_Protocol" title="User Datagram Protocol">User Datagram Protocol</a> at the <a href="/wiki/Transport_layer" title="Transport layer">transport layer</a> and the Internet Protocol at the <a href="/wiki/Internet_layer" title="Internet layer">internet layer</a>. The model became known as the <i>Department of Defense (DoD) Internet Model</i> and <i><a href="/wiki/Internet_protocol_suite" title="Internet protocol suite">Internet protocol suite</a></i>, and informally as <i>TCP/IP</i>.
</p><p>The following <a href="/wiki/Internet_Experiment_Note" title="Internet Experiment Note">Internet Experiment Note</a> (IEN) documents describe the evolution of the Internet Protocol into the modern version of IPv4:<sup id="cite&#95;ref-6" class="reference"><a href="#cite_note-6"><span class="cite-bracket">&#91;</span>6<span class="cite-bracket">&#93;</span></a></sup>
</p>
<ul><li><a rel="nofollow" class="external text" href="http://www.rfc-editor.org/ien/ien2.txt">IEN 2</a> <i>Comments on Internet Protocol and TCP (</i>August 1977) describes the need to separate the TCP and Internet Protocol functionalities (which were previously combined).  It proposes the first version of the IP header, using 0 for the version field.</li>
<li><a rel="nofollow" class="external text" href="http://www.rfc-editor.org/ien/ien26.pdf">IEN 26</a> <i>A Proposed New Internet Header Format (</i>February 1978) describes a version of the IP header that uses a 1-bit version field.</li>
<li><a rel="nofollow" class="external text" href="http://www.rfc-editor.org/ien/ien28.pdf">IEN 28</a> <i>Draft Internetwork Protocol Description Version 2 (</i>February 1978) describes IPv2.</li>
<li><a rel="nofollow" class="external text" href="http://www.rfc-editor.org/ien/ien41.pdf">IEN 41</a> <i>Internetwork Protocol Specification Version 4 (</i>June 1978) describes the first protocol to be called IPv4.  The IP header is different from the modern IPv4 header.</li>
<li><a rel="nofollow" class="external text" href="http://www.rfc-editor.org/ien/ien44.pdf">IEN 44</a> <i>Latest Header Formats (</i>June 1978) describes another version of IPv4, also with a header different from the modern IPv4 header.</li>
<li><a rel="nofollow" class="external text" href="http://www.rfc-editor.org/ien/ien54.pdf">IEN 54</a> <i>Internetwork Protocol Specification Version 4 (</i>September 1978) is the first description of <p>IP versions 1 to 3 were experimental versions, designed between 1973 and 1978.<sup id="cite&#95;ref-Coty&#95;7-0" class="reference"><a href="#cite_note-Coty-7"><span class="cite-bracket">&#91;</span>7<span class="cite-bracket">&#93;</span></a></sup> Versions 2 and 3 supported variable-length addresses ranging between 1 and 16 octets (between 8 and 128 bits).<sup id="cite&#95;ref-8" class="reference"><a href="#cite_note-8"><span class="cite-bracket">&#91;</span>8<span class="cite-bracket">&#93;</span></a></sup> An early draft of version 4 supported variable-length addresses of up to 256 octets (up to 2048 bits)<sup id="cite&#95;ref-9" class="reference"><a href="#cite_note-9"><span class="cite-bracket">&#91;</span>9<span class="cite-bracket">&#93;</span></a></sup> but this was later abandoned in favor of a fixed-size 32-bit address in the final version of <a href="/wiki/IPv4" title="IPv4">IPv4</a>. This remains the dominant internetworking protocol in use in the <a href="/wiki/Internet_Layer" class="mw-redirect" title="Internet Layer">Internet Layer</a>; the number 4 identifies the protocol version, carried in every IP datagram. IPv4 is defined in <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r1333433106" /><a href="/wiki/RFC_(identifier)" class="mw-redirect" title="RFC (identifier)">RFC</a>&#160;<a rel="nofollow" class="external text" href="https://www.rfc-editor.org/rfc/rfc791">791</a> (1981).
</p><p>Version number 5 was used by the <a href="/wiki/Internet_Stream_Protocol" title="Internet Stream Protocol">Internet Stream Protocol</a>, an experimental streaming protocol that was not adopted.<sup id="cite&#95;ref-Coty&#95;7-1" class="reference"><a href="#cite_note-Coty-7"><span class="cite-bracket">&#91;</span>7<span class="cite-bracket">&#93;</span></a></sup>
</p><p>The successor to IPv4 is <a href="/wiki/IPv6" title="IPv6">IPv6</a>. IPv6 was a result of several years of experimentation and dialog during which various protocol models were proposed, such as TP/IX (<link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r1333433106" /><a href="/wiki/RFC_(identifier)" class="mw-redirect" title="RFC (identifier)">RFC</a>&#160;<a rel="nofollow" class="external text" href="https://www.rfc-editor.org/rfc/rfc1475">1475</a>), PIP (<link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r1333433106" /><a href="/wiki/RFC_(identifier)" class="mw-redirect" title="RFC (identifier)">RFC</a>&#160;<a rel="nofollow" class="external text" href="https://www.rfc-editor.org/rfc/rfc1621">1621</a>) and TUBA (TCP and UDP with Bigger Addresses, <link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r1333433106" /><a href="/wiki/RFC_(identifier)" class="mw-redirect" title="RFC (identifier)">RFC</a>&#160;<a rel="nofollow" class="external text" href="https://www.rfc-editor.org/rfc/rfc1347">1347</a>). Its most prominent difference from version 4 is the size of the addresses. While IPv4 uses <a href="/wiki/32_bits" class="mw-redirect" title="32 bits">32 bits</a> for addressing, yielding c. 4.3 <a href="/wiki/1,000,000,000_(number)" class="mw-redirect" title="1,000,000,000 (number)">billion</a> (<span class="nowrap"><span data-sort-value="7009430000000000000♠"></span>4.3<span style="margin-left:0.25em;margin-right:0.15em;">×</span>10<sup>9</sup></span>) addresses, IPv6 uses <a href="/wiki/128-bit" class="mw-redirect" title="128-bit">128-bit</a> addresses providing c. <span class="nowrap"><span data-sort-value="7038340000000000000♠"></span>3.4<span style="margin-left:0.25em;margin-right:0.15em;">×</span>10<sup>38</sup></span> addresses. Although adoption of IPv6 has been slow, as of January&#160;2023<sup class="plainlinks noexcerpt noprint asof-tag update" style="display:none;"><a class="external text" href="https://en.wikipedia.org/w/index.php?title=Internet_Protocol&amp;action=edit">&#91;update&#93;</a></sup>, most countries in the world show significant adoption of IPv6,<sup id="cite&#95;ref-10" class="reference"><a href="#cite_note-10"><span class="cite-bracket">&#91;</span>10<span class="cite-bracket">&#93;</span></a></sup> with over 41% of Google's traffic being carried over IPv6 connections.<sup id="cite&#95;ref-11" class="reference"><a href="#cite_note-11"><span class="cite-bracket">&#91;</span>11<span class="cite-bracket">&#93;</span></a></sup>
</p><p>The assignment of the new protocol as IPv6 was uncertain until due diligence assured that IPv6 had not been used previously.<sup id="cite&#95;ref-12" class="reference"><a href="#cite_note-12"><span class="cite-bracket">&#91;</span>12<span class="cite-bracket">&#93;</span></a></sup> Other Internet Layer protocols have been assigned version numbers,<sup id="cite&#95;ref-13" class="reference"><a href="#cite_note-13"><span class="cite-bracket">&#91;</span>13<span class="cite-bracket">&#93;</span></a></sup> such as 7 (<i>IP/TX</i>), 8 and 9 (<i>historic</i>). Notably, on April 1, 1994, the <a href="/wiki/IETF" class="mw-redirect" title="IETF">IETF</a> published an <a href="/wiki/April_Fools%27_Day_RfC" class="mw-redirect" title="April Fools&#39; Day RfC">April Fools' Day RfC</a> about IPv9.<sup id="cite&#95;ref-14" class="reference"><a href="#cite_note-14"><span class="cite-bracket">&#91;</span>14<span class="cite-bracket">&#93;</span></a></sup> IPv9 was also used in an alternate proposed address space expansion called TUBA.<sup id="cite&#95;ref-15" class="reference"><a href="#cite_note-15"><span class="cite-bracket">&#91;</span>15<span class="cite-bracket">&#93;</span></a></sup> A 2004 Chinese proposal for <a href="/wiki/IPv9_(China)" title="IPv9 (China)">an IPv9 protocol</a> appears to be unrelated to all of these, and is not endorsed by the IETF.
</p>

<div class="mw-heading mw-heading2"><h2 id="Reliability">Reliability</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Internet_Protocol&amp;action=edit&amp;section=5" title="Edit section: Reliability"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>The design of the Internet protocol suite adheres to the <a href="/wiki/End-to-end_principle" title="End-to-end principle">end-to-end principle</a>, a concept adapted from the <a href="/wiki/CYCLADES" title="CYCLADES">CYCLADES</a> project. Under the end-to-end principle, the network infrastructure is considered inherently unreliable at any single network element or transmission medium and is dynamic in terms of the availability of links and nodes. No central monitoring or performance measurement facility exists that tracks or maintains the state of the network. For the benefit of reducing <a href="/wiki/Network_complexity" title="Network complexity">network complexity</a>, the intelligence in the network is located in the <a href="/wiki/End_node" class="mw-redirect" title="End node">end nodes</a>.
</p><p>As a consequence of this design, the Internet Protocol only provides <a href="/wiki/Best-effort_delivery" title="Best-effort delivery">best-effort delivery</a> and its service is characterized as <a href="/wiki/Reliability_(computer_networking)" title="Reliability (computer networking)">unreliable</a>. In network architectural parlance, it is a <a href="/wiki/Connectionless_protocol" class="mw-redirect" title="Connectionless protocol">connectionless protocol</a>, in contrast to <a href="/wiki/Connection-oriented_communication" title="Connection-oriented communication">connection-oriented communication</a>. Various fault conditions may occur, such as <a href="/wiki/Data_corruption" title="Data corruption">data corruption</a>, <a href="/wiki/Packet_loss" title="Packet loss">packet loss</a> and duplication. Because routing is dynamic, meaning every packet is treated independently, and because the network maintains no state based on the path of prior packets, different packets may be routed to the same destination via different paths, resulting in <a href="/wiki/Out-of-order_delivery" title="Out-of-order delivery">out-of-order delivery</a> to the receiver.
</p><p>All fault conditions in the network must be detected and compensated for by the participating end nodes. The <a href="/wiki/Upper_layer_protocol" class="mw-redirect" title="Upper layer protocol">upper layer protocols</a> of the Internet protocol suite are responsible for resolving reliability issues. For example, a host may <a href="/wiki/Data_buffer" title="Data buffer">buffer</a> network data to ensure correct ordering before the data is delivered to an application.
</p><p>IPv4 provides safeguards to ensure that the header of an IP packet is error-free. A routing node discards packets that fail a header <a href="/wiki/Checksum" title="Checksum">checksum</a> test. Although the <a href="/wiki/Internet_Control_Message_Protocol" title="Internet Control Message Protocol">Internet Control Message Protocol</a> (ICMP) provides notification of errors, a routing node is not required to notify either end node of errors. IPv6, by contrast, operates without header checksums, since current <a href="/wiki/Link_layer" title="Link layer">link layer</a> technology is assumed to provide sufficient error detection.<sup id="cite&#95;ref-rfc1726&#95;25-0" class="reference"><a href="#cite_note-rfc1726-25"><span class="cite-bracket">&#91;</span>25<span class="cite-bracket">&#93;</span></a></sup><sup class="reference nowrap"><span title="Location: §6.2">&#58;&#8202;§6.2&#8202;</span></sup><sup id="cite&#95;ref-rfc8200&#95;19-1" class="reference"><a href="#cite_note-rfc8200-19"><span class="cite-bracket">&#91;</span>19<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading2"><h2 id="Link_capacity_and_capability">Link capacity and capability</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Internet_Protocol&amp;action=edit&amp;section=6" title="Edit section: Link capacity and capability"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>The dynamic nature of the Internet and the diversity of its components provide no guarantee that any particular path is actually capable of, or suitable for, performing the data transmission requested. One of the technical constraints is the size of data packets possible on a given link. Facilities exist to examine the <a href="/wiki/Maximum_transmission_unit" title="Maximum transmission unit">maximum transmission unit</a> (MTU) size of the local link and <a href="/wiki/Path_MTU_Discovery" title="Path MTU Discovery">Path MTU Discovery</a> can be used for the entire intended path to the destination.<sup id="cite&#95;ref-26" class="reference"><a href="#cite_note-26"><span class="cite-bracket">&#91;</span>26<span class="cite-bracket">&#93;</span></a></sup>
</p><p>The IPv4 internetworking layer automatically <a href="/wiki/IP_fragmentation" title="IP fragmentation">fragments</a> a datagram into smaller units for transmission when the link MTU is exceeded. IP provides re-ordering of fragments received out of order.<sup id="cite&#95;ref-27" class="reference"><a href="#cite_note-27"><span class="cite-bracket">&#91;</span>27<span class="cite-bracket">&#93;</span></a></sup> An IPv6 network does not perform fragmentation in network elements, but requires end hosts and higher-layer protocols to avoid exceeding the path MTU.<sup id="cite&#95;ref-28" class="reference"><a href="#cite_note-28"><span class="cite-bracket">&#91;</span>28<span class="cite-bracket">&#93;</span></a></sup>
</p><p>The <a href="/wiki/Transmission_Control_Protocol" title="Transmission Control Protocol">Transmission Control Protocol</a> (TCP) is an example of a protocol that adjusts its segment size to be smaller than the MTU. The <a href="/wiki/User_Datagram_Protocol" title="User Datagram Protocol">User Datagram Protocol</a> (UDP) and ICMP disregard MTU size, thereby forcing IP to fragment oversized datagrams.<sup id="cite&#95;ref-29" class="reference"><a href="#cite_note-29"><span class="cite-bracket">&#91;</span>29<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading2"><h2 id="Security">Security</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Internet_Protocol&amp;action=edit&amp;section=7" title="Edit section: Security"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>During the design phase of the <a href="/wiki/ARPANET" title="ARPANET">ARPANET</a> and the early Internet, the security aspects and needs of a public, international network were not adequately anticipated.  Consequently, many Internet protocols exhibited vulnerabilities highlighted by network attacks and later security assessments. In 2008, a thorough security assessment and proposed mitigation of problems was published.<sup id="cite&#95;ref-30" class="reference"><a href="#cite_note-30"><span class="cite-bracket">&#91;</span>30<span class="cite-bracket">&#93;</span></a></sup> The IETF has been pursuing further studies.<sup id="cite&#95;ref-&#82;FC&#95;6274&#95;31-0" class="reference"><a href="#cite_note-RFC_6274-31"><span class="cite-bracket">&#91;</span>31<span class="cite-bracket">&#93;</span></a></sup>
</p>
