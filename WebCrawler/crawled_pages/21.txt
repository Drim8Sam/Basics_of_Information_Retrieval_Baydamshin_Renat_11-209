<p><b>Memory management</b> (also <b>dynamic memory management</b>, <b>dynamic storage allocation</b>, or <b>dynamic memory allocation</b>) is a form of <a href="/wiki/Resource_management_(computing)" title="Resource management (computing)">resource management</a> applied to <a href="/wiki/Computer_memory" title="Computer memory">computer memory</a>. The essential requirement of memory management is to provide ways to dynamically allocate portions of memory to programs at their request, and free it for reuse when no longer needed. This is critical to any advanced computer system where more than a single <a href="/wiki/Process_(computing)" title="Process (computing)">process</a> might be underway (<a href="/wiki/Computer_multitasking" title="Computer multitasking">multitasking</a>) at any time.<sup id="cite&#95;ref-:0&#95;1-0" class="reference"><a href="#cite_note-:0-1"><span class="cite-bracket">&#91;</span>1<span class="cite-bracket">&#93;</span></a></sup> 
</p><p>Several methods have been devised that increase the effectiveness of memory management. <a href="/wiki/Virtual_memory" title="Virtual memory">Virtual memory</a> systems separate the <a href="/wiki/Memory_address" title="Memory address">memory addresses</a> used by a process from actual physical addresses, allowing separation of processes and increasing the size of the <a href="/wiki/Virtual_address_space" title="Virtual address space">virtual address space</a> beyond the available amount of <a href="/wiki/Random-access_memory" title="Random-access memory">RAM</a> using <a href="/wiki/Paging" class="mw-redirect" title="Paging">paging</a> or swapping to <a href="/wiki/Secondary_storage" class="mw-redirect" title="Secondary storage">secondary storage</a>. The quality of the virtual memory manager can have an extensive effect on overall system <a href="/wiki/Computer_performance" title="Computer performance">performance</a>. The system allows a computer to appear as if it may have more memory available than physically present, thereby allowing multiple processes to share it. 
</p><p>In some <a href="/wiki/Operating_system" title="Operating system">operating systems</a>, e.g. <a href="/wiki/Burroughs_MCP" title="Burroughs MCP">Burroughs/Unisys MCP</a>,<sup id="cite&#95;ref-Unisys-MCP-Memory&#95;2-0" class="reference"><a href="#cite_note-Unisys-MCP-Memory-2"><span class="cite-bracket">&#91;</span>2<span class="cite-bracket">&#93;</span></a></sup> and <a href="/wiki/OS/360_and_successors" title="OS/360 and successors">OS/360 and successors</a>,<sup id="cite&#95;ref-3" class="reference"><a href="#cite_note-3"><span class="cite-bracket">&#91;</span>3<span class="cite-bracket">&#93;</span></a></sup> memory is managed by the operating system.<sup id="cite&#95;ref-4" class="reference"><a href="#cite_note-4"><span class="cite-bracket">&#91;</span>note 1<span class="cite-bracket">&#93;</span></a></sup> In other operating systems, e.g. <a href="/wiki/Unix-like" title="Unix-like">Unix-like</a> operating systems, memory is managed at the application level.
</p><p>Memory management within an address space is generally categorized as either <a href="/wiki/Manual_memory_management" title="Manual memory management">manual memory management</a> or automatic memory management.
</p>
<meta property="mw:PageProp/toc" />
<div class="mw-heading mw-heading2"><h2 id="Manual_memory_management"><span class="anchor" id="HEAP"></span> Manual memory management</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Memory_management&amp;action=edit&amp;section=1" title="Edit section: Manual memory management"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r1320445320" /><div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Manual_memory_management" title="Manual memory management">Manual memory management</a></div>
<figure typeof="mw:File/Thumb"><a href="/wiki/File:External_Fragmentation.svg" class="mw-file-description"><img src="//upload.wikimedia.org/wikipedia/commons/thumb/4/4a/External_Fragmentation.svg/500px-External_Fragmentation.svg.png" decoding="async" width="450" height="270" class="mw-file-element" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/4/4a/External_Fragmentation.svg/960px-External_Fragmentation.svg.png 1.5x" data-file-width="700" data-file-height="420" /></a><figcaption>An example of external fragmentation</figcaption></figure>
<p>The task of fulfilling an allocation request consists of locating a block of unused memory of sufficient size. Memory requests are satisfied by allocating portions from a large pool<sup id="cite&#95;ref-5" class="reference"><a href="#cite_note-5"><span class="cite-bracket">&#91;</span>note 2<span class="cite-bracket">&#93;</span></a></sup> of memory called the <i>heap</i><sup id="cite&#95;ref-6" class="reference"><a href="#cite_note-6"><span class="cite-bracket">&#91;</span>note 3<span class="cite-bracket">&#93;</span></a></sup> or <i>free store</i>. At any given time, some parts of the heap are in use, while some are "free" (unused) and thus available for future allocations.
In the C language, the function which allocates memory from the heap is called <code class="mw-highlight mw-highlight-lang-text mw-content-ltr" style="" dir="ltr">malloc</code> and the function which takes previously allocated memory and marks it as "free" (to be used by future allocations) is called <code class="mw-highlight mw-highlight-lang-text mw-content-ltr" style="" dir="ltr">free</code>.<sup id="cite&#95;ref-8" class="reference"><a href="#cite_note-8"><span class="cite-bracket">&#91;</span>note 4<span class="cite-bracket">&#93;</span></a></sup>
</p><p>Several issues complicate the implementation, such as <a href="/wiki/Fragmentation_(computer)#External_fragmentation" class="mw-redirect" title="Fragmentation (computer)">external fragmentation</a>, which arises when there are many small gaps between allocated memory blocks, which invalidates their use for an allocation request. The allocator's <a href="/wiki/Metadata_(computing)" class="mw-redirect" title="Metadata (computing)">metadata</a> can also inflate the size of (individually) small allocations. This is often managed by <a href="/wiki/Chunking_(computing)" title="Chunking (computing)">chunking</a>. The memory management system must track outstanding allocations to ensure that they do not overlap and that no memory is ever "lost" (i.e. that there are no "<a href="/wiki/Memory_leak" title="Memory leak">memory leaks</a>").
</p>
<div class="mw-heading mw-heading3"><h3 id="Efficiency">Efficiency</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Memory_management&amp;action=edit&amp;section=2" title="Edit section: Efficiency"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>The specific dynamic memory allocation algorithm implemented can impact performance significantly. A study conducted in 1994 by <a href="/wiki/Digital_Equipment_Corporation" title="Digital Equipment Corporation">Digital Equipment Corporation</a> illustrates the <a href="/wiki/Computational_overhead" class="mw-redirect" title="Computational overhead">overheads</a> involved for a variety of allocators. The lowest average <a href="/wiki/Instruction_path_length" title="Instruction path length">instruction path length</a> required to allocate a single memory slot was 52 (as measured with an instruction level <a href="/wiki/Profiling_(computer_programming)" title="Profiling (computer programming)">profiler</a> on a variety of software).<sup id="cite&#95;ref-:0&#95;1-1" class="reference"><a href="#cite_note-:0-1"><span class="cite-bracket">&#91;</span>1<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading3"><h3 id="Implementations">Implementations</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Memory_management&amp;action=edit&amp;section=3" title="Edit section: Implementations"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>Since the precise location of the allocation is not known in advance, the memory is accessed indirectly, usually through a <a href="/wiki/Pointer_(computer_programming)" title="Pointer (computer programming)">pointer</a> <a href="/wiki/Reference_(computer_science)" title="Reference (computer science)">reference</a>. The specific algorithm used to organize the memory area and allocate and deallocate chunks is interlinked with the <a href="/wiki/Kernel_(operating_system)" title="Kernel (operating system)">kernel</a>, and may use any of the following methods:
</p>
<div class="mw-heading mw-heading4"><h4 id="Fixed-size_blocks_allocation"><span class="anchor" id="FIXED-SIZE"></span>Fixed-size blocks allocation</h4><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Memory_management&amp;action=edit&amp;section=4" title="Edit section: Fixed-size blocks allocation"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r1320445320" /><div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Memory_pool" title="Memory pool">Memory pool</a></div>
<p>Fixed-size blocks allocation, also called memory pool allocation, uses a <a href="/wiki/Free_list" title="Free list">free list</a> of fixed-size blocks of memory (often all of the same size). This works well for simple <a href="/wiki/Embedded_system" title="Embedded system">embedded systems</a> where no large objects need to be allocated but suffers from <a href="/wiki/Fragmentation_(computing)" title="Fragmentation (computing)">fragmentation</a> especially with long memory addresses. However, due to the significantly reduced overhead, this method can substantially improve performance for objects that need frequent allocation and deallocation, and so it is often used in <a href="/wiki/Video_games" class="mw-redirect" title="Video games">video games</a>.
</p>
<div class="mw-heading mw-heading4"><h4 id="Buddy_blocks">Buddy blocks</h4><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Memory_management&amp;action=edit&amp;section=5" title="Edit section: Buddy blocks"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r1320445320" /><div role="note" class="hatnote navigation-not-searchable">Further information: <a href="/wiki/Buddy_memory_allocation" title="Buddy memory allocation">Buddy memory allocation</a></div>
<p>In this system, memory is allocated into several pools of memory instead of just one, where each pool represents blocks of memory of a certain <a href="/wiki/Power_of_two" title="Power of two">power of two</a> in size, or blocks of some other convenient size progression. All blocks of a particular size are kept in a sorted <a href="/wiki/Linked_list" title="Linked list">linked list</a> or <a href="/wiki/Tree_data_structure" class="mw-redirect" title="Tree data structure">tree</a> and all new blocks that are formed during allocation are added to their respective memory pools for later use. If a smaller size is requested than is available, the smallest available size is selected and split. One of the resulting parts is selected, and the process repeats until the request is complete. When a block is allocated, the allocator will start with the smallest sufficiently large block to avoid needlessly breaking blocks. When a block is freed, it is compared to its buddy. If they are both free, they are combined and placed in the correspondingly larger-sized buddy-block list.
</p>
<div class="mw-heading mw-heading4"><h4 id="Slab_allocation">Slab allocation</h4><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Memory_management&amp;action=edit&amp;section=6" title="Edit section: Slab allocation"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r1320445320" /><div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Slab_allocation" title="Slab allocation">Slab allocation</a></div>
<p>This memory allocation mechanism preallocates memory chunks suitable to fit objects of a certain type or size.<sup id="cite&#95;ref-silberschatz&#95;9-0" class="reference"><a href="#cite_note-silberschatz-9"><span class="cite-bracket">&#91;</span>5<span class="cite-bracket">&#93;</span></a></sup> These chunks are called caches and the allocator only has to keep track of a list of free cache slots. Constructing an object will use any one of the free cache slots and destructing an object will add a slot back to the free cache slot list. This technique alleviates memory fragmentation and is efficient as there is no need to search for a suitable portion of memory, as any open slot will suffice.
</p>
<div class="mw-heading mw-heading4"><h4 id="Stack_allocation">Stack allocation</h4><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Memory_management&amp;action=edit&amp;section=7" title="Edit section: Stack allocation"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r1320445320" /><div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Stack-based_memory_allocation" title="Stack-based memory allocation">Stack-based memory allocation</a></div>
<p>Many <a href="/wiki/Unix-like" title="Unix-like">Unix-like</a> systems as well as <a href="/wiki/Microsoft_Windows" title="Microsoft Windows">Microsoft Windows</a> implement a function called <code class="mw-highlight mw-highlight-lang-text mw-content-ltr" style="" dir="ltr">alloca</code> for dynamically allocating stack memory in a way similar to the heap-based <code class="mw-highlight mw-highlight-lang-text mw-content-ltr" style="" dir="ltr">malloc</code>. A compiler typically translates it to inlined instructions manipulating the stack pointer.<sup id="cite&#95;ref-10" class="reference"><a href="#cite_note-10"><span class="cite-bracket">&#91;</span>6<span class="cite-bracket">&#93;</span></a></sup> Although there is no need of manually freeing memory allocated this way as it is automatically freed when the function that called <code class="mw-highlight mw-highlight-lang-text mw-content-ltr" style="" dir="ltr">alloca</code> returns, there exists a risk of overflow. And since alloca is an <i>ad hoc</i> expansion seen in many systems but never in <a href="/wiki/POSIX" title="POSIX">POSIX</a> or the C standard, its behavior in case of a stack overflow is undefined.
</p><p>A safer version of alloca called <code class="mw-highlight mw-highlight-lang-text mw-content-ltr" style="" dir="ltr">_malloca</code>, which reports errors, exists on Microsoft Windows. It requires the use of <code class="mw-highlight mw-highlight-lang-text mw-content-ltr" style="" dir="ltr">_freea</code>.<sup id="cite&#95;ref-11" class="reference"><a href="#cite_note-11"><span class="cite-bracket">&#91;</span>7<span class="cite-bracket">&#93;</span></a></sup> <a href="/wiki/Gnulib" title="Gnulib">gnulib</a> provides an equivalent interface, albeit instead of throwing an SEH exception on overflow, it delegates to malloc when an overlarge size is detected.<sup id="cite&#95;ref-12" class="reference"><a href="#cite_note-12"><span class="cite-bracket">&#91;</span>8<span class="cite-bracket">&#93;</span></a></sup> A similar feature can be emulated using manual accounting and size-checking, such as in the uses of <code class="mw-highlight mw-highlight-lang-text mw-content-ltr" style="" dir="ltr">alloca_account</code> in glibc.<sup id="cite&#95;ref-13" class="reference"><a href="#cite_note-13"><span class="cite-bracket">&#91;</span>9<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading2"><h2 id="Automated_memory_management">Automated memory management</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Memory_management&amp;action=edit&amp;section=8" title="Edit section: Automated memory management"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>The proper management of memory in an application is a difficult problem, and several different strategies for handling memory management have been devised.
</p>
<div class="mw-heading mw-heading3"><h3 id="Automatic_management_of_call_stack_variables">Automatic management of call stack variables</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Memory_management&amp;action=edit&amp;section=9" title="Edit section: Automatic management of call stack variables"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r1320445320" /><div role="note" class="hatnote navigation-not-searchable">See also: <a href="/wiki/Automatic_variable" title="Automatic variable">Automatic variable</a> and <a href="/wiki/Call_stack" title="Call stack">Call stack</a></div>
<p>In many programming language implementations, the runtime environment for the program automatically allocates memory in the <a href="/wiki/Call_stack" title="Call stack">call stack</a> for non-static <a href="/wiki/Local_variable" title="Local variable">local variables</a> of a <a href="/wiki/Subroutine" class="mw-redirect" title="Subroutine">subroutine</a>, called <a href="/wiki/Automatic_variable" title="Automatic variable">automatic variables</a>, when the subroutine is called, and automatically releases that memory when the subroutine is exited. Special declarations may allow local variables to retain values between invocations of the procedure, or may allow local variables to be accessed by other subroutines. The automatic allocation of local variables makes <a href="/wiki/Recursion_(computer_science)" title="Recursion (computer science)">recursion</a> possible, to a depth limited by available memory.
</p>
<div class="mw-heading mw-heading3"><h3 id="Garbage_collection">Garbage collection</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Memory_management&amp;action=edit&amp;section=10" title="Edit section: Garbage collection"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r1320445320" /><div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Garbage_collection_(computer_science)" title="Garbage collection (computer science)">Garbage collection (computer science)</a></div>
<p>Garbage collection is a strategy for automatically detecting memory allocated to objects that are no longer usable in a program, and returning that allocated memory to a pool of free memory locations. This method is in contrast to "manual" memory management where a programmer explicitly codes memory requests and memory releases in the program. While automatic garbage collection has the advantages of reducing programmer workload and preventing certain kinds of memory allocation bugs, garbage collection does require memory resources of its own, and can compete with the application program for processor time.
</p>
<div class="mw-heading mw-heading3"><h3 id="Reference_counting">Reference counting</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Memory_management&amp;action=edit&amp;section=11" title="Edit section: Reference counting"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r1320445320" /><div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Reference_counting" title="Reference counting">Reference counting</a></div>
<p>Reference counting is a strategy for detecting that memory is no longer usable by a program by maintaining a counter for how many independent pointers point to the memory. Whenever a new pointer points to a piece of memory, the programmer is supposed to increase the counter. When the pointer changes where it points, or when the pointer is no longer pointing to any area or has itself been freed, the counter should decrease. When the counter drops to zero, the memory should be considered unused and freed. Some <a href="/wiki/Reference_counting" title="Reference counting">reference counting</a> systems require programmer involvement and some are implemented automatically by the compiler. A disadvantage of reference counting is that <a href="/wiki/Circular_reference" title="Circular reference">circular references</a> can develop which cause a memory leak to occur. This can be mitigated by either adding the concept of a "<a href="/wiki/Weak_reference" title="Weak reference">weak reference</a>" (a reference that does not participate in reference counting, but is notified when the area it is pointing to is no longer valid) or by combining reference counting and garbage collection together.
</p>
<div class="mw-heading mw-heading3"><h3 id="Memory_pools">Memory pools</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Memory_management&amp;action=edit&amp;section=12" title="Edit section: Memory pools"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r1320445320" /><div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Region-based_memory_management" title="Region-based memory management">Region-based memory management</a></div>
<p>A memory pool is a technique of automatically deallocating memory based on the state of the application, such as the lifecycle of a request or transaction. The idea is that many applications execute large chunks of code which may generate memory allocations, but that there is a point in execution where all of those chunks are known to be no longer valid. For example, in a web service, after each request the web service no longer needs any of the memory allocated during the execution of the request. Therefore, rather than keeping track of whether or not memory is currently being referenced, the memory is allocated according to the request or lifecycle stage with which it is associated. When that request or stage has passed, all associated memory is deallocated simultaneously.
</p>
<div class="mw-heading mw-heading2"><h2 id="Systems_with_virtual_memory">Systems with virtual memory</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Memory_management&amp;action=edit&amp;section=13" title="Edit section: Systems with virtual memory"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r1320445320" /><div role="note" class="hatnote navigation-not-searchable">Main articles: <a href="/wiki/Memory_protection" title="Memory protection">Memory protection</a>, <a href="/wiki/Virtual_Memory" class="mw-redirect" title="Virtual Memory">Virtual Memory</a>, and <a href="/wiki/Shared_memory_(interprocess_communication)" class="mw-redirect" title="Shared memory (interprocess communication)">Shared memory (interprocess communication)</a></div>
<p><a href="/wiki/Virtual_memory" title="Virtual memory">Virtual memory</a> is a method of decoupling the memory organization from the physical hardware. The applications operate on memory via <i>virtual addresses</i>. Each attempt by the application to access a particular virtual memory address results in the virtual memory address being translated to an actual <i>physical address</i>.<sup id="cite&#95;ref-14" class="reference"><a href="#cite_note-14"><span class="cite-bracket">&#91;</span>10<span class="cite-bracket">&#93;</span></a></sup>  In this way the addition of virtual memory enables granular control over memory systems and methods of access.
</p><p>In virtual memory systems the operating system limits how a <a href="/wiki/Process_(computing)" title="Process (computing)">process</a> can access the memory. This feature, called <a href="/wiki/Memory_protection" title="Memory protection">memory protection</a>, can be used to disallow a process to read or write to memory that is not allocated to it, preventing malicious or malfunctioning code in one program from interfering with the operation of another.
</p><p>Even though the memory allocated for specific processes is normally isolated, processes sometimes need to be able to share information. <a href="/wiki/Shared_memory_(interprocess_communication)" class="mw-redirect" title="Shared memory (interprocess communication)">Shared memory</a> is one of the fastest techniques for <a href="/wiki/Inter-process_communication" title="Inter-process communication">inter-process communication</a>.
</p><p>Memory is usually classified by access rate into <a href="/wiki/Primary_storage" class="mw-redirect" title="Primary storage">primary storage</a> and <a href="/wiki/Secondary_storage" class="mw-redirect" title="Secondary storage">secondary storage</a>. Memory management systems, among other operations, also handle the moving of information between these two levels of memory.
</p>
<div class="mw-heading mw-heading2"><h2 id="Memory_management_in_Burroughs/Unisys_MCP_systems[2]"><span id="Memory_management_in_Burroughs.2FUnisys_MCP_systems.5B2.5D"></span>Memory management in Burroughs/Unisys MCP systems<sup id="cite&#95;ref-Unisys-MCP-Memory&#95;2-1" class="reference"><a href="#cite_note-Unisys-MCP-Memory-2"><span class="cite-bracket">&#91;</span>2<span class="cite-bracket">&#93;</span></a></sup></h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Memory_management&amp;action=edit&amp;section=14" title="Edit section: Memory management in Burroughs/Unisys MCP systems[2]"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>An operating system manages various resources in the computing system. The memory subsystem is the system element for managing memory. The memory subsystem combines the hardware memory resource and the MCP OS software that manages the resource.
</p><p>The memory subsystem manages the physical memory and the virtual memory of the system (both part of the hardware resource). The virtual memory extends physical memory by using extra space on a peripheral device, usually disk. The memory subsystem is responsible for moving code and data between main and virtual memory in a process known as overlaying. Burroughs was the first commercial implementation of virtual memory (although developed at Manchester University for the Ferranti Atlas computer) and integrated virtual memory with the system design of the B5000 from the start (in 1961) needing no external <a href="/wiki/Memory_management_unit" title="Memory management unit">memory management unit</a> (MMU).<sup id="cite&#95;ref-Waychoff&#95;1979&#95;15-0" class="reference"><a href="#cite_note-Waychoff_1979-15"><span class="cite-bracket">&#91;</span>11<span class="cite-bracket">&#93;</span></a></sup><sup class="reference nowrap"><span title="Page / location: 48">&#58;&#8202;48&#8202;</span></sup>
</p><p>The memory subsystem is responsible for mapping logical requests for memory blocks to physical portions of memory (segments) which are found in the list of free segments. Each allocated block is managed by means of a segment descriptor,<sup id="cite&#95;ref-MCP&#95;Descriptor&#95;16-0" class="reference"><a href="#cite_note-MCP_Descriptor-16"><span class="cite-bracket">&#91;</span>12<span class="cite-bracket">&#93;</span></a></sup> a special control word containing relevant metadata about the segment including address, length, machine type, and the p-bit or ‘presence’ bit which indicates whether the block is in main memory or needs to be loaded from the address given in the descriptor.
</p><p><a href="/wiki/Burroughs_large_systems_descriptors" title="Burroughs large systems descriptors">Descriptors</a> are essential in providing memory safety and security so that operations cannot overflow or underflow the referenced block (commonly known as buffer overflow). Descriptors themselves are protected control words that cannot be manipulated except for specific elements of the MCP OS (enabled by the UNSAFE block directive in <a href="/wiki/NEWP" class="mw-redirect" title="NEWP">NEWP</a>).
</p><p>Donald Knuth describes a similar system in Section 2.5 ‘Dynamic Storage Allocation’ of <a href="/wiki/The_Art_of_Computer_Programming" title="The Art of Computer Programming">‘Fundamental Algorithms’</a>.<sup class="noprint Inline-Template" style="white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:Disputed_statement" class="mw-redirect" title="Wikipedia:Disputed statement"><span title="This claim has reliable sources with contradicting facts (November 2024)">disputed</span></a>&#32;&#8211; <a class="mw-selflink-fragment" href="#Burroughs/Unisys_MCP_memory_management_system_discussed_in_Knuth?">discuss</a></i>&#93;</sup>
</p>
<div class="mw-heading mw-heading2"><h2 id="Memory_management_in_OS/360_and_successors"><span id="Memory_management_in_OS.2F360_and_successors"></span>Memory management in OS/360 and successors</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Memory_management&amp;action=edit&amp;section=15" title="Edit section: Memory management in OS/360 and successors"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>IBM <a href="/wiki/System/360" class="mw-redirect" title="System/360">System/360</a> does not support virtual memory.<sup id="cite&#95;ref-17" class="reference"><a href="#cite_note-17"><span class="cite-bracket">&#91;</span>note 5<span class="cite-bracket">&#93;</span></a></sup> Memory isolation of <a href="/wiki/Job_(computing)" title="Job (computing)">jobs</a> is optionally accomplished using <a href="/wiki/Memory_protection#Protection_keys" title="Memory protection">protection keys</a>, assigning storage for each job a different key, 0 for the supervisor or 1–15. Memory management in <a href="/wiki/OS/360_and_successors" title="OS/360 and successors">OS/360</a> is a <a href="/wiki/Supervisory_program" title="Supervisory program">supervisor</a> function. Storage is requested using the <code>GETMAIN</code> macro and freed using the <code>FREEMAIN</code> macro, which result in a call to the supervisor (<a href="/wiki/Supervisor_Call_instruction" title="Supervisor Call instruction">SVC</a>) to perform the operation.
</p><p>In OS/360 the details vary depending on how the system is <a href="/wiki/System_generation" title="System generation">generated</a>, e.g., for <a href="/wiki/OS/360_and_successors#PCP" title="OS/360 and successors">PCP</a>, <a href="/wiki/OS/360_and_successors#MFT" title="OS/360 and successors">MFT</a>, <a href="/wiki/OS/360_and_successors#MVT" title="OS/360 and successors">MVT</a>.
</p><p>In OS/360 MVT, suballocation within a job's <i>region</i> or the shared <i>System Queue Area</i> (SQA) is based on <i>subpools</i>, areas a multiple of 2&#160;KB in size—the size of an area protected by a protection key. Subpools are numbered 0–255.<sup id="cite&#95;ref-FOOTNOTEOS360Sup&#91;httpbitsaversorgpdfibm360osR217&#95;Apr73GC28-6646-7&#95;Supervisor&#95;Services&#95;and&#95;Macro&#95;Instructions&#95;Rel&#95;217&#95;Sep74pdfpage100&#95;82&#93;–85&#95;18-0" class="reference"><a href="#cite_note-FOOTNOTEOS360Sup[httpbitsaversorgpdfibm360osR217_Apr73GC28-6646-7_Supervisor_Services_and_Macro_Instructions_Rel_217_Sep74pdfpage100_82]–85-18"><span class="cite-bracket">&#91;</span>13<span class="cite-bracket">&#93;</span></a></sup> Within a region subpools are assigned either the job's storage protection or the supervisor's key, key 0. Subpools 0–127 receive the job's key. Initially only subpool zero is created, and all user storage requests are satisfied from subpool 0, unless another is specified in the memory request. Subpools 250–255 are created by memory requests by the supervisor on behalf of the job. Most of these are assigned key 0, although a few get the key of the job. Subpool numbers are also relevant in MFT, although the details are much simpler.<sup id="cite&#95;ref-FOOTNOTEOS360Sup&#91;httpbitsaversorgpdfibm360osR217&#95;Apr73GC28-6646-7&#95;Supervisor&#95;Services&#95;and&#95;Macro&#95;Instructions&#95;Rel&#95;217&#95;Sep74pdfpage100&#95;82&#93;&#95;19-0" class="reference"><a href="#cite_note-FOOTNOTEOS360Sup[httpbitsaversorgpdfibm360osR217_Apr73GC28-6646-7_Supervisor_Services_and_Macro_Instructions_Rel_217_Sep74pdfpage100_82]-19"><span class="cite-bracket">&#91;</span>14<span class="cite-bracket">&#93;</span></a></sup>  MFT uses fixed <i>partitions</i> redefinable by the operator instead of dynamic regions and PCP has only a single partition.
</p><p>Each subpool is mapped by a list of control blocks identifying allocated and free memory blocks within the subpool. Memory is allocated by finding a free area of sufficient size, or by allocating additional blocks in the subpool, up to the region size of the job. It is possible to free all or part of an allocated memory area.<sup id="cite&#95;ref-SupvrLogic&#95;20-0" class="reference"><a href="#cite_note-SupvrLogic-20"><span class="cite-bracket">&#91;</span>15<span class="cite-bracket">&#93;</span></a></sup>
</p><p>The details for <a href="/wiki/OS/VS1" title="OS/VS1">OS/VS1</a> are similar<sup id="cite&#95;ref-FOOTNOTEOSVS1Dig&#91;httpsbitsaversorgpdfibm370OS&#95;VS1GC24-5091-5&#95;OS&#95;VS1&#95;Release&#95;6&#95;Programmers&#95;Reference&#95;Digest&#95;197609pdfpage114&#95;p.&#95;2.37-2.39&#93;&#95;21-0" class="reference"><a href="#cite_note-FOOTNOTEOSVS1Dig[httpsbitsaversorgpdfibm370OS_VS1GC24-5091-5_OS_VS1_Release_6_Programmers_Reference_Digest_197609pdfpage114_p._2.37-2.39]-21"><span class="cite-bracket">&#91;</span>16<span class="cite-bracket">&#93;</span></a></sup>  to those for MFT and for MVT; the details for <a href="/wiki/OS/VS2" title="OS/VS2">OS/VS2</a> are similar to those for MVT, except that the page size is 4 KiB. For both OS/VS1 and OS/VS2 the shared <i>System Queue Area</i> (SQA) is nonpageable.
</p><p>In <a href="/wiki/MVS" title="MVS">MVS</a> the address space<sup id="cite&#95;ref-22" class="reference"><a href="#cite_note-22"><span class="cite-bracket">&#91;</span>17<span class="cite-bracket">&#93;</span></a></sup> includes an additional pageable shared area, the <i>Common Storage Area</i> (CSA), and two additional private areas, the nonpageable <i>local system queue area</i> (LSQA) and the pageable <i>System Work area</i> (SWA). Also, the storage keys 0–7 are all reserved for use by privileged code.
</p>
