					<div id="mw-content-text" class="mw-body-content"><div class="mw-subjectpageheader">
</div><div class="mw-content-ltr mw-parser-output" lang="en" dir="ltr"><div class="shortdescription nomobile noexcerpt noprint searchaux" style="display:none">High-level computer programming conceptualization</div>
<style data-mw-deduplicate="TemplateStyles:r1320445320">.mw-parser-output .hatnote{font-style:italic}.mw-parser-output div.hatnote{padding-left:1.6em;margin-bottom:0.5em}.mw-parser-output .hatnote i{font-style:normal}.mw-parser-output .hatnote+link+.hatnote{margin-top:-0.5em}@media print{body.ns-0 .mw-parser-output .hatnote{display:none!important}}</style><div role="note" class="hatnote navigation-not-searchable">This article is about classification of programming languages. For definition of the term "programming model", see <a href="/wiki/Programming_model" title="Programming model">Programming model</a>.</div>
<p>A <b>programming paradigm</b> is a relatively high-level way to conceptualize and structure the implementation of a <a href="/wiki/Computer_program" title="Computer program">computer program</a>. A <a href="/wiki/Programming_language" title="Programming language">programming language</a> can be classified as supporting one or many paradigms.<sup id="cite&#95;ref-1" class="reference"><a href="#cite_note-1"><span class="cite-bracket">&#91;</span>1<span class="cite-bracket">&#93;</span></a></sup>
</p><p>Paradigms are separated along and described by different dimensions of programming. Some paradigms are about implications of the <a href="/wiki/Execution_model" title="Execution model">execution model</a>, such as allowing <a href="/wiki/Side_effect_(computer_science)" title="Side effect (computer science)">side effects</a>, or whether the sequence of operations is defined by the execution model. Other paradigms are about the way code is organized, such as grouping into units that include both state and behavior. Yet others are about <a href="/wiki/Syntax_(programming_languages)" title="Syntax (programming languages)">syntax</a> and <a href="/wiki/Formal_grammar" title="Formal grammar">grammar</a>.
</p><p>Some common programming paradigms include (shown in <a href="/wiki/Hierarchical_relationship" class="mw-redirect" title="Hierarchical relationship">hierarchical relationship</a>):<sup id="cite&#95;ref-2" class="reference"><a href="#cite_note-2"><span class="cite-bracket">&#91;</span>2<span class="cite-bracket">&#93;</span></a></sup><sup id="cite&#95;ref-3" class="reference"><a href="#cite_note-3"><span class="cite-bracket">&#91;</span>3<span class="cite-bracket">&#93;</span></a></sup><sup id="cite&#95;ref-symbolic-programming-lisp&#95;4-0" class="reference"><a href="#cite_note-symbolic-programming-lisp-4"><span class="cite-bracket">&#91;</span>4<span class="cite-bracket">&#93;</span></a></sup>
</p>
<ul><li><a href="/wiki/Imperative_programming" title="Imperative programming">Imperative</a> – code directly controls <a href="/wiki/Control_flow" title="Control flow">execution flow</a> and state change, explicit statements that change a program state
<ul><li><a href="/wiki/Procedural_programming" title="Procedural programming">procedural</a> – organized as <a href="/wiki/Function_(computer_programming)" title="Function (computer programming)">procedures</a> that call each other</li>
<li><a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented</a> – organized as <a href="/wiki/Object_(computer_science)" title="Object (computer science)">objects</a> that contain both data structure and associated behavior, uses data structures consisting of data fields and methods together with their interactions (objects) to design programs
<ul><li><a href="/wiki/Class_(programming)" title="Class (programming)">Class-based</a> – object-oriented programming in which <a href="/wiki/Inheritance_(object-oriented_programming)" title="Inheritance (object-oriented programming)">inheritance</a> is achieved by defining classes of objects, versus the objects themselves</li>
<li><a href="/wiki/Object-based_programming" class="mw-redirect" title="Object-based programming">Object-based</a> - paradigm which the object has a construct to encapsulate state and behavior.</li>
<li><a href="/wiki/Prototype-based_programming" title="Prototype-based programming">Prototype-based</a> – object-oriented programming that avoids classes and implements inheritance via cloning of instances</li></ul></li></ul></li>
<li><a href="/wiki/Declarative_programming" title="Declarative programming">Declarative</a> – code declares properties of the desired result, but not how to compute it, describes what computation should perform, without specifying detailed state changes
<ul><li><a href="/wiki/Functional_programming" title="Functional programming">functional</a> – a desired result is declared as the value of a series of function evaluations, uses evaluation of mathematical functions and avoids state and mutable data</li>
<li><a href="/wiki/Logic_programming" title="Logic programming">logic</a> – a desired result is declared as the answer to a question about a system of facts and rules, uses explicit mathematical logic for programming</li>
<li><a href="/wiki/Reactive_programming" title="Reactive programming">reactive</a> – a desired result is declared with data streams and the propagation of change</li></ul></li>
<li><a href="/wiki/Concurrent_programming_language" class="mw-redirect" title="Concurrent programming language">Concurrent programming</a> – has language constructs for concurrency, these may involve multi-threading, support for distributed computing, message passing, shared resources (including shared memory), or <a href="/wiki/Futures_and_promises" title="Futures and promises">futures</a>
<ul><li><a href="/wiki/Actor_model" title="Actor model">Actor programming</a> – concurrent computation with <i>actors</i> that make local decisions in response to the environment (capable of selfish or competitive behaviour)</li></ul></li>
<li><a href="/wiki/Constraint_programming" title="Constraint programming">Constraint programming</a> – relations between variables are expressed as constraints (or constraint networks), directing allowable solutions (uses constraint satisfaction or <a href="/wiki/Simplex_algorithm" title="Simplex algorithm">simplex algorithm</a>)</li>
<li><a href="/wiki/Dataflow" title="Dataflow">Dataflow programming</a> – forced recalculation of formulas when data values change (e.g. <a href="/wiki/Spreadsheet" title="Spreadsheet">spreadsheets</a>)</li>
<li><a href="/wiki/Distributed_computing" title="Distributed computing">Distributed programming</a> – has support for multiple autonomous computers that communicate via computer networks</li>
<li><a href="/wiki/Generic_programming" title="Generic programming">Generic programming</a> – uses algorithms written in terms of to-be-specified-later types that are then instantiated as needed for specific types provided as parameters</li>
<li><a href="/wiki/Metaprogramming" title="Metaprogramming">Metaprogramming</a> – writing programs that write or manipulate other programs (or themselves) as their data, or that do part of the work at compile time that would otherwise be done at runtime
<ul><li><a href="/wiki/Template_metaprogramming" title="Template metaprogramming">Template metaprogramming</a> – metaprogramming methods in which a compiler uses templates to generate temporary source code, which is merged by the compiler with the rest of the source code and then compiled</li>
<li><a href="/wiki/Reflective_programming" title="Reflective programming">Reflective programming</a> – metaprogramming methods in which a program modifies or extends itself</li></ul></li>
<li><a href="/wiki/Pipeline_programming" class="mw-redirect" title="Pipeline programming">Pipeline programming</a> – a simple syntax change to add syntax to nest function calls to language originally designed with none</li>
<li><a href="/wiki/Rule-based_programming" class="mw-redirect" title="Rule-based programming">Rule-based programming</a> – a network of rules of thumb that comprise a knowledge base and can be used for expert systems and problem deduction &amp; resolution</li>
<li><a href="/wiki/Visual_programming_language" title="Visual programming language">Visual programming</a> – manipulating program elements graphically rather than by specifying them textually (e.g. <a href="/wiki/Simulink" title="Simulink">Simulink</a>); also termed <i>diagrammatic programming</i></li></ul>

<div class="mw-heading mw-heading2"><h2 id="Overview">Overview</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Programming_paradigm&amp;action=edit&amp;section=1" title="Edit section: Overview"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<figure class="mw-halign-left" typeof="mw:File/Thumb"><a href="/wiki/File:Programming_paradigms.svg" class="mw-file-description"><img src="//upload.wikimedia.org/wikipedia/commons/thumb/f/f7/Programming_paradigms.svg/250px-Programming_paradigms.svg.png" decoding="async" width="250" height="217" class="mw-file-element" srcset="//upload.wikimedia.org/wikipedia/commons/thumb/f/f7/Programming_paradigms.svg/500px-Programming_paradigms.svg.png 1.5x" data-file-width="2868" data-file-height="2491" /></a><figcaption>Overview of the various programming paradigms according to Peter Van Roy<sup id="cite&#95;ref-5" class="reference"><a href="#cite_note-5"><span class="cite-bracket">&#91;</span>5<span class="cite-bracket">&#93;</span></a></sup><sup class="reference nowrap"><span title="Page / location: 5">&#58;&#8202;5&#8202;</span></sup><sup id="cite&#95;ref-Van-RoyHaridi2004&#95;6-0" class="reference"><a href="#cite_note-Van-RoyHaridi2004-6"><span class="cite-bracket">&#91;</span>6<span class="cite-bracket">&#93;</span></a></sup></figcaption></figure>
<p>Programming paradigms come from <a href="/wiki/Computer_science" title="Computer science">computer science</a> <a href="/wiki/Research" title="Research">research</a> into existing practices of <a href="/wiki/Software_development" title="Software development">software development</a>. The findings allow for describing and comparing programming practices and the languages used to code programs. For perspective, other fields of research study 
<a href="/wiki/Software_engineering" title="Software engineering">software engineering</a> <a href="/wiki/Process" title="Process">processes</a> and describe various <a href="/wiki/Methodology" title="Methodology">methodologies</a> to describe and compare them.
</p><p>A <a href="/wiki/Programming_language" title="Programming language">programming language</a> can be described in terms of paradigms. Some languages support only one paradigm. For example, <a href="/wiki/Smalltalk" title="Smalltalk">Smalltalk</a> supports object-oriented and <a href="/wiki/Haskell" title="Haskell">Haskell</a> supports functional. Most languages support multiple paradigms. For example, a program written in <a href="/wiki/C%2B%2B" title="C++">C++</a>, <a href="/wiki/Object_Pascal" title="Object Pascal">Object Pascal</a>, or <a href="/wiki/PHP" title="PHP">PHP</a> can be purely <a href="/wiki/Procedural_programming" title="Procedural programming">procedural</a>, purely <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented</a>, or can contain aspects of both paradigms, or others. 
</p><p>When using a language that supports multiple paradigms, the developer chooses which paradigm elements to use. But, this choice may not involve considering paradigms per se. The developer often uses the features of a language as the language provides them and to the extent that the developer knows them. Categorizing the resulting <a href="/wiki/Source_code" title="Source code">code</a> by paradigm is often an academic activity done in retrospect.
</p><p>Languages categorized as <b>imperative paradigm</b> have two main features: they state the order in which operations occur, with constructs that explicitly control that order, and they allow side effects, in which state can be modified at one point in time, within one unit of code, and then later read at a different point in time inside a different unit of code.  The communication between the units of code is not explicit.
</p><p>In contrast, languages in the <b>declarative paradigm</b> do not state the order in which to execute operations. Instead, they supply a number of available operations in the system, along with the conditions under which each is allowed to execute.<sup id="cite&#95;ref-7" class="reference"><a href="#cite_note-7"><span class="cite-bracket">&#91;</span>7<span class="cite-bracket">&#93;</span></a></sup>  The implementation of the language's execution model tracks which operations are free to execute and chooses the order independently. More at  <a href="/wiki/Comparison_of_multi-paradigm_programming_languages" title="Comparison of multi-paradigm programming languages">Comparison of multi-paradigm programming languages</a>.
</p><p>In <b>object-oriented</b> programming, code is organized into <a href="/wiki/Object_(programming)" class="mw-redirect" title="Object (programming)">objects</a> that contain state that is owned by and (usually) controlled by the code of the object.  Most object-oriented languages are also imperative languages.
</p><p>In object-oriented programming, programs are treated as a set of interacting objects. In <a href="/wiki/Functional_programming" title="Functional programming">functional programming</a>, programs are treated as a sequence of stateless function evaluations. When programming computers or systems with many processors, in <a href="/wiki/Process-oriented_programming" title="Process-oriented programming">process-oriented programming</a>, programs are treated as sets of concurrent processes that act on a logical shared <a href="/wiki/Data_structure" title="Data structure">data structures</a>.
</p><p>Many programming paradigms are as well known for the techniques they <i>forbid</i> as for those they <i>support</i>. For instance, pure functional programming disallows <a href="/wiki/Side-effect_(computer_science)" class="mw-redirect" title="Side-effect (computer science)">side-effects</a>, while <a href="/wiki/Structured_programming" title="Structured programming">structured programming</a> disallows the <a href="/wiki/Goto" title="Goto">goto</a> construct. Partly for this reason, new paradigms are often regarded as doctrinaire or overly rigid by those accustomed to older ones.<sup id="cite&#95;ref-rubin87goto&#95;8-0" class="reference"><a href="#cite_note-rubin87goto-8"><span class="cite-bracket">&#91;</span>8<span class="cite-bracket">&#93;</span></a></sup> Yet, avoiding certain techniques can make it easier to understand program behavior, and to <a href="/wiki/Automated_theorem_proving" title="Automated theorem proving">prove theorems</a> about program correctness.
</p><p>Programming paradigms can also be compared with <i><a href="/wiki/Programming_model" title="Programming model">programming models</a></i>, which allows invoking an <a href="/wiki/Execution_model" title="Execution model">execution model</a> by using only an API. Programming models can also be classified into paradigms based on features of the execution model.
</p><p>For <a href="/wiki/Parallel_computing" title="Parallel computing">parallel computing</a>, using a programming model instead of a language is common.  The reason is that details of the parallel hardware leak into the abstractions used to program the hardware.  This causes the programmer to have to map patterns in the algorithm onto patterns in the execution model (which have been inserted due to leakage of hardware into the abstraction).  As a consequence, no one parallel programming language maps well to all computation problems.  Thus, it is more convenient to use a base sequential language and insert API calls to parallel execution models via a programming model.  Such parallel programming models can be classified according to abstractions that reflect the hardware, such as <a href="/wiki/Shared_memory" title="Shared memory">shared memory</a>, <a href="/wiki/Distributed_memory" title="Distributed memory">distributed memory</a> with <a href="/wiki/Message_passing" title="Message passing">message passing</a>, notions of <i>place</i> visible in the code, and so forth.  These can be considered flavors of programming paradigm that apply to only parallel languages and programming models.
</p>
<div class="mw-heading mw-heading2"><h2 id="Criticism">Criticism</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Programming_paradigm&amp;action=edit&amp;section=2" title="Edit section: Criticism"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>Some programming language researchers criticise the notion of paradigms as a classification of programming languages, e.g. Harper,<sup id="cite&#95;ref-9" class="reference"><a href="#cite_note-9"><span class="cite-bracket">&#91;</span>9<span class="cite-bracket">&#93;</span></a></sup> and Krishnamurthi.<sup id="cite&#95;ref-10" class="reference"><a href="#cite_note-10"><span class="cite-bracket">&#91;</span>10<span class="cite-bracket">&#93;</span></a></sup>  They argue that many programming languages cannot be strictly classified into one paradigm, but rather include features from several paradigms. See <a href="/wiki/Comparison_of_multi-paradigm_programming_languages" title="Comparison of multi-paradigm programming languages">Comparison of multi-paradigm programming languages</a>.
</p>
<div class="mw-heading mw-heading2"><h2 id="History">History</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Programming_paradigm&amp;action=edit&amp;section=3" title="Edit section: History"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>Different approaches to programming have developed over time. Classification of each approach was either described at the time the approach was first developed, but often not until some time later, retrospectively. An early approach consciously identified as such is <a href="/wiki/Structured_programming" title="Structured programming">structured programming</a>, advocated since the mid 1960s. The concept of a <i>programming paradigm</i> as such dates at least to 1978, in the <a href="/wiki/Turing_Award" title="Turing Award">Turing Award</a> lecture of <a href="/wiki/Robert_W._Floyd" title="Robert W. Floyd">Robert W. Floyd</a>, entitled <i>The Paradigms of Programming</i>, which cites the notion of paradigm as used by <a href="/wiki/Thomas_Kuhn" title="Thomas Kuhn">Thomas Kuhn</a> in his <i><a href="/wiki/The_Structure_of_Scientific_Revolutions" title="The Structure of Scientific Revolutions">The Structure of Scientific Revolutions</a></i> (1962).<sup id="cite&#95;ref-11" class="reference"><a href="#cite_note-11"><span class="cite-bracket">&#91;</span>11<span class="cite-bracket">&#93;</span></a></sup> Early programming languages did not have clearly defined programming paradigms and sometimes programs made extensive use of goto statements, liberal use of which led to <a href="/wiki/Spaghetti_code" title="Spaghetti code">spaghetti code</a> which is difficult to understand and maintain. This led to the development of structured programming paradigms that disallowed the use of goto statements, only allowing the use of more structured programming constructs.<sup id="cite&#95;ref-12" class="reference"><a href="#cite_note-12"><span class="cite-bracket">&#91;</span>12<span class="cite-bracket">&#93;</span></a></sup>
</p>
<div class="mw-heading mw-heading2"><h2 id="Languages_and_paradigms">Languages and paradigms</h2><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Programming_paradigm&amp;action=edit&amp;section=4" title="Edit section: Languages and paradigms"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<div class="mw-heading mw-heading3"><h3 id="Machine_code">Machine code</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Programming_paradigm&amp;action=edit&amp;section=5" title="Edit section: Machine code"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p><a href="/wiki/Machine_code" title="Machine code">Machine code</a> is the lowest-level of computer programming as it is <a href="/wiki/Instruction_set" class="mw-redirect" title="Instruction set">machine instructions</a> that define behavior at the lowest level of abstract possible for a computer. As it is the most prescriptive way to code it is classified as imperative.
</p><p>It is sometimes called the <a href="/wiki/First-generation_programming_language" title="First-generation programming language">first-generation programming language</a>.
</p>
<div class="mw-heading mw-heading3"><h3 id="Assembly">Assembly</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Programming_paradigm&amp;action=edit&amp;section=6" title="Edit section: Assembly"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p><a href="/wiki/Assembly_language" title="Assembly language">Assembly language</a> introduced mnemonics for machine instructions and <a href="/wiki/Memory_address" title="Memory address">memory addresses</a>. Assembly is classified as imperative and is sometimes called the <a href="/wiki/Second-generation_programming_language" title="Second-generation programming language">second-generation programming language</a>.
</p><p>In the 1960s, assembly languages were developed to support library COPY and quite sophisticated conditional macro generation and preprocessing abilities, CALL to <a href="/wiki/Subroutine" class="mw-redirect" title="Subroutine">subroutine</a>, external variables and common sections (globals), enabling significant code re-use and isolation from hardware specifics via the use of logical operators such as READ/WRITE/GET/PUT. Assembly was, and still is, used for time-critical systems and often in <a href="/wiki/Embedded_system" title="Embedded system">embedded systems</a> as it gives the most control of what the machine does.
</p>
<div class="mw-heading mw-heading3"><h3 id="Procedural_languages">Procedural languages</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Programming_paradigm&amp;action=edit&amp;section=7" title="Edit section: Procedural languages"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p><a href="/wiki/Procedural_language" class="mw-redirect" title="Procedural language">Procedural languages</a>, also called the <a href="/wiki/Third-generation_programming_language" title="Third-generation programming language">third-generation programming languages</a> are the first described as <a href="/wiki/High-level_programming_language" title="High-level programming language">high-level languages</a>. They support vocabulary related to the problem being solved. For example,
</p>
<ul><li>COmmon Business Oriented Language (<a href="/wiki/COBOL" title="COBOL">COBOL</a>)&#160;&#8211;&#32; uses terms like <a href="/wiki/Computer_file" title="Computer file">file</a>, <a href="/wiki/Move_(command)" title="Move (command)">move</a> and <a href="/wiki/Copy_(command)" title="Copy (command)">copy</a>.</li>
<li>FORmula TRANslation (<a href="/wiki/FORTRAN" class="mw-redirect" title="FORTRAN">FORTRAN</a>)&#160;&#8211;&#32; using <a href="/wiki/Mathematical" class="mw-redirect" title="Mathematical">mathematical</a> language terminology, it was developed mainly for scientific and engineering problems.</li>
<li>ALGOrithmic Language (<a href="/wiki/ALGOL" title="ALGOL">ALGOL</a>)&#160;&#8211;&#32; focused on being an appropriate language to define <a href="/wiki/Algorithm" title="Algorithm">algorithms</a>, while using mathematical language terminology, targeting scientific and engineering problems, just like FORTRAN.</li>
<li>Programming Language One (<a href="/wiki/PL/I" title="PL/I">PL/I</a>)&#160;&#8211;&#32; a hybrid commercial-scientific general purpose language supporting <a href="/wiki/Pointer_(computer_programming)" title="Pointer (computer programming)">pointers</a>.</li>
<li>Beginners All purpose Symbolic Instruction Code (<a href="/wiki/BASIC" title="BASIC">BASIC</a>)&#160;&#8211;&#32; it was developed to enable more people to write programs.</li>
<li><a href="/wiki/C_(programming_language)" title="C (programming language)">C</a>&#160;&#8211;&#32; a general-purpose programming language, initially developed by <a href="/wiki/Dennis_Ritchie" title="Dennis Ritchie">Dennis Ritchie</a> between 1969 and 1973 at <a href="/wiki/AT%26T_Bell_Labs" class="mw-redirect" title="AT&amp;T Bell Labs">AT&amp;T Bell Labs</a>.</li></ul>
<p>These languages are classified as procedural paradigm. They directly control the step by step process that a computer program follows. The <a href="/wiki/Efficacy" title="Efficacy">efficacy</a> and <a href="/wiki/Algorithmic_efficiency" title="Algorithmic efficiency">efficiency</a> of such a program is therefore highly dependent on the programmer's skill.
</p>
<div class="mw-heading mw-heading3"><h3 id="Object-oriented_programming">Object-oriented programming</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Programming_paradigm&amp;action=edit&amp;section=8" title="Edit section: Object-oriented programming"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<link rel="mw-deduplicated-inline-style" href="mw-data:TemplateStyles:r1320445320" /><div role="note" class="hatnote navigation-not-searchable">Main article: <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">Object-oriented programming</a></div>
<p>In attempt to improve on procedural languages, <a href="/wiki/Object-oriented_programming" title="Object-oriented programming">object-oriented programming</a> (OOP) languages were created, such as <a href="/wiki/Simula" title="Simula">Simula</a>, <a href="/wiki/Smalltalk" title="Smalltalk">Smalltalk</a>, <a href="/wiki/C%2B%2B" title="C++">C++</a>, <a href="/wiki/Eiffel_(programming_language)" title="Eiffel (programming language)">Eiffel</a>, <a href="/wiki/Python_(programming_language)" title="Python (programming language)">Python</a>, <a href="/wiki/PHP" title="PHP">PHP</a>, <a href="/wiki/Java_(programming_language)" title="Java (programming language)">Java</a>, and <a href="/wiki/C_Sharp_(programming_language)" title="C Sharp (programming language)">C#</a>. In these languages, <a href="/wiki/Data" title="Data">data</a> and methods to manipulate the data are in the same code unit called an <a href="/wiki/Object_(computer_science)" title="Object (computer science)">object</a>. This <a href="/wiki/Encapsulation_(computer_programming)" title="Encapsulation (computer programming)">encapsulation</a> ensures that the only way that an object can access data is via <i><a href="/wiki/Method_(computer_programming)" title="Method (computer programming)">methods</a></i> of the object that contains the data. Thus, an object's inner workings may be changed without affecting code that uses the object. 
</p><p>There is <a href="/wiki/Object-oriented_programming#Criticism" title="Object-oriented programming">controversy</a> raised by <a href="/wiki/Alexander_Stepanov" title="Alexander Stepanov">Alexander Stepanov</a>, <a href="/wiki/Richard_Stallman" title="Richard Stallman">Richard Stallman</a><sup id="cite&#95;ref-13" class="reference"><a href="#cite_note-13"><span class="cite-bracket">&#91;</span>13<span class="cite-bracket">&#93;</span></a></sup> and other programmers, concerning the efficacy of the OOP paradigm versus the procedural paradigm. The need for every object to have associative methods leads some skeptics to associate OOP with <a href="/wiki/Software_bloat" title="Software bloat">software bloat</a>; an attempt to resolve this dilemma came through <a href="/wiki/Polymorphism_(computer_science)" title="Polymorphism (computer science)">polymorphism</a>.
</p><p>Although most OOP languages are third-generation, it is possible to create an object-oriented assembler language. <a href="/wiki/High_Level_Assembly" title="High Level Assembly">High Level Assembly</a> (HLA) is an example of this that fully supports advanced data types and object-oriented assembly language programming&#160;&#8211;&#32; despite its early origins. Thus, differing programming paradigms can be seen rather like <i>motivational <a href="/wiki/Meme" title="Meme">memes</a></i> of their advocates, rather than necessarily representing progress from one level to the next.<sup class="noprint Inline-Template Template-Fact" style="white-space:nowrap;">&#91;<i><a href="/wiki/Wikipedia:Citation_needed" title="Wikipedia:Citation needed"><span title="This claim needs references to reliable sources. (March 2018)">citation needed</span></a></i>&#93;</sup> Precise comparisons of competing paradigms' efficacy are frequently made more difficult because of new and differing terminology applied to similar entities and processes together with numerous implementation distinctions across languages.
</p>
<div class="mw-heading mw-heading3"><h3 id="Declarative_languages">Declarative languages</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Programming_paradigm&amp;action=edit&amp;section=9" title="Edit section: Declarative languages"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p>A <a href="/wiki/Declarative_programming" title="Declarative programming">declarative programming</a> program describes what the problem is, not how to solve it. The program is structured as a set of properties to find in the expected result, not as a procedure to follow. Given a database or a set of rules, the computer tries to find a solution matching all the desired properties. An archetype of a declarative language is the <a href="/wiki/Fourth-generation_programming_language" title="Fourth-generation programming language">fourth generation language</a> <a href="/wiki/SQL" title="SQL">SQL</a>, and the family of functional languages and logic programming.
</p><p><a href="/wiki/Functional_programming" title="Functional programming">Functional programming</a> is a subset of declarative programming. Programs written using this paradigm use <a href="/wiki/Subroutine" class="mw-redirect" title="Subroutine">functions</a>, blocks of code intended to behave like <a href="/wiki/Function_(mathematics)" title="Function (mathematics)">mathematical functions</a>. Functional languages discourage changes in the value of variables through <a href="/wiki/Assignment_(computer_science)" title="Assignment (computer science)">assignment</a>, making a great deal of use of <a href="/wiki/Recursion_(computer_science)" title="Recursion (computer science)">recursion</a> instead.
</p><p>The <a href="/wiki/Logic_programming" title="Logic programming">logic programming</a> paradigm views computation as <a href="/wiki/Automated_reasoning" title="Automated reasoning">automated reasoning</a> over a body of knowledge. Facts about the <a href="/wiki/Domain_(software_engineering)" title="Domain (software engineering)">problem domain</a> are expressed as logic formulas, and programs are executed by applying <a href="/wiki/Inference_rule" class="mw-redirect" title="Inference rule">inference rules</a> over them until an answer to the problem is found, or the set of formulas is proved inconsistent.
</p>
<div class="mw-heading mw-heading3"><h3 id="Other_paradigms">Other paradigms</h3><span class="mw-editsection"><span class="mw-editsection-bracket">[</span><a href="/w/index.php?title=Programming_paradigm&amp;action=edit&amp;section=10" title="Edit section: Other paradigms"><span>edit</span></a><span class="mw-editsection-bracket">]</span></span></div>
<p><a href="/wiki/Symbolic_programming" title="Symbolic programming">Symbolic programming</a> is a paradigm that describes programs able to manipulate formulas and program components as data.<sup id="cite&#95;ref-symbolic-programming-lisp&#95;4-1" class="reference"><a href="#cite_note-symbolic-programming-lisp-4"><span class="cite-bracket">&#91;</span>4<span class="cite-bracket">&#93;</span></a></sup> Programs can thus effectively modify themselves, and appear to "learn", making them suited for applications such as <a href="/wiki/Artificial_intelligence" title="Artificial intelligence">artificial intelligence</a>, <a href="/wiki/Expert_system" title="Expert system">expert systems</a>, <a href="/wiki/Natural-language_processing" class="mw-redirect" title="Natural-language processing">natural-language processing</a> and computer games. Languages that support this paradigm include <a href="/wiki/Lisp_(programming_language)" title="Lisp (programming language)">Lisp</a> and <a href="/wiki/Prolog" title="Prolog">Prolog</a>.<sup id="cite&#95;ref-14" class="reference"><a href="#cite_note-14"><span class="cite-bracket">&#91;</span>14<span class="cite-bracket">&#93;</span></a></sup>
</p><p><a href="/wiki/Differentiable_programming" title="Differentiable programming">Differentiable programming</a> structures programs so that they can be <a href="/wiki/Differentiation_(mathematics)" class="mw-redirect" title="Differentiation (mathematics)">differentiated</a> throughout, usually via <a href="/wiki/Automatic_differentiation" title="Automatic differentiation">automatic differentiation</a>.<sup id="cite&#95;ref-15" class="reference"><a href="#cite_note-15"><span class="cite-bracket">&#91;</span>15<span class="cite-bracket">&#93;</span></a></sup><sup id="cite&#95;ref-innes&#95;16-0" class="reference"><a href="#cite_note-innes-16"><span class="cite-bracket">&#91;</span>16<span class="cite-bracket">&#93;</span></a></sup>
</p><p><a href="/wiki/Literate_programming" title="Literate programming">Literate programming</a>, as a form of <a href="/wiki/Imperative_programming" title="Imperative programming">imperative programming</a>, structures programs as a human-centered web, as in a <a href="/wiki/Hypertext" title="Hypertext">hypertext</a> essay: documentation is integral to the program, and the program is structured following the logic of prose exposition, rather than compiler convenience.
</p><p><a href="/wiki/Symbolic_programming" title="Symbolic programming">Symbolic programming</a> techniques such as <a href="/wiki/Reflective_programming" title="Reflective programming">reflective programming</a> (reflection), which allow a program to refer to itself, might also be considered as a programming paradigm. However, this is compatible with the major paradigms and thus is not a real paradigm in its own right.
</p>
